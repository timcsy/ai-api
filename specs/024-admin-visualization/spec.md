# 功能規格：Admin 視覺化強化

**Feature Branch**: `024-admin-visualization`
**建立日期**: 2026-06-03
**狀態**: Draft
**輸入**: 使用者描述:「直接做完整個階段 14——首頁與用量頁加入決策導向圖表、導入第一個圖表 lib、平台級 daily 時序、暫停/隔離原因顯眼化。」

## 使用者情境與測試 *(mandatory)*

### 使用者故事 1 - 管理員一眼掌握平台用量節奏（優先級：P1）

管理員打開管理首頁，除了既有的設定狀態與隔離警示，現在看到三張決策導向圖表：(1) 過去 7–14 天的每日花費／用量長條圖（可切 token / 花費）；(2) 各 model 花費占比的環圈圖；(3) 花最多的前 5 個分配。他不必點進用量頁逐項翻，就能回答「最近花錢節奏正常嗎？錢燒在哪個 model？誰是重度使用者？」。圖表放在隔離警示之下，警示仍最顯眼。

**為何此優先級**：這是本階段的招牌價值——把平台已有的可追蹤資料變成 admin「一眼可決策」的形式，且這是競品 mockup 與「像成品」觀感差距最大的地方。

**獨立可測**：seed 跨 model / 跨分配的呼叫，開首頁，驗證三張圖數字正確、quarantine 警示仍在最上方。

**驗收情境**：

1. **Given** 平台過去兩週有每日呼叫，**When** admin 開首頁，**Then** 看到 daily spend 長條圖，每日一根、hover 顯示當日明細，可切換 token / 花費兩種度量
2. **Given** 用量分散在多個 model，**When** admin 看首頁，**Then** Spend by Model 環圈圖顯示 top 5 model + 「其他」，點一塊跳該 model 的用量詳情
3. **Given** 有多筆分配，**When** admin 看首頁，**Then** Top 5 分配長條圖顯示花最多的 5 筆（友善名 + 數字），點一筆跳該分配詳情
4. **Given** 有分配被自動隔離，**When** admin 開首頁，**Then** 隔離警示卡仍在所有圖表**之上**、最顯眼處（圖表不得淹沒它）
5. **Given** 首頁載入，**When** 渲染完成，**Then** 圖表數量 ≤ 3 張

---

### 使用者故事 2 - 管理員在用量頁做更深入的維度分析（優先級：P2）

管理員到用量頁想了解「平台對哪家 provider 依賴最重」與「使用集中在哪些時段」。他看到兩個新視覺：(1) Provider 占比環圈圖（Azure / OpenAI / Anthropic / Gemini 比例）；(2) 24 小時 × 7 天的使用熱力圖（哪個時段最熱）。後者對課堂場景特別有意義——可看出上課時間的尖峰。

**為何此優先級**：用量頁是 admin 做 procurement / 排程決策的地方；provider 占比影響採購、時段熱力圖影響容量規劃。但比首頁三圖次要（首頁是日常入口）。

**獨立可測**：seed 跨 provider / 跨時段的呼叫，開用量頁，驗證兩個視覺正確。

**驗收情境**：

1. **Given** 用量分散在多家 provider，**When** admin 看用量頁，**Then** Provider 占比環圈圖顯示各家比例
2. **Given** 呼叫集中在特定時段，**When** admin 看熱力圖，**Then** 該時段格子顏色較深，可辨識尖峰
3. **Given** 熱力圖呈現，**When** admin 檢視，**Then** 軸標清楚（24 小時 × 星期幾），對非技術 admin 可讀

---

### 使用者故事 3 - 管理員用統一時段選擇器調整所有圖表範圍（優先級：P2）

管理員想把首頁與用量頁的圖表都切到「本季」來看趨勢。每個有圖表的頁面右上有統一的時段選擇器（本週 / 本月 / 本季 / 自訂），切換後該頁所有圖表一起更新。

**為何此優先級**：沒有時段控制，圖表只能看固定區間，決策力受限；但它是圖表的輔助控制，非圖表本身。

**驗收情境**：

1. **Given** 首頁有時段選擇器，**When** admin 切到「本季」，**Then** 該頁所有圖表更新為本季資料
2. **Given** 選「自訂」，**When** admin 指定起訖日，**Then** 圖表更新為該區間
3. **Given** 切換時段，**When** 更新中，**Then** 有載入指示，不是空白閃爍

---

### 使用者故事 4 - 管理員一眼看到分配為何被暫停/隔離（優先級：P2）

管理員在分配列表看到一筆「已隔離」的分配，把滑鼠移到徽章上（或點進解除頁），直接看到觸發原因與具體數據——例如「過去 1 小時 1100 calls，baseline 100/hr，超出 11×」。他不必再點進稽核紀錄翻找。

**為何此優先級**：目前隔離/暫停的原因藏在稽核紀錄裡，admin 要處理時得多翻一層；顯眼化大幅降低處理摩擦。屬「維運可視性」的收尾。

**驗收情境**：

1. **Given** 一筆分配被異常偵測器隔離，**When** admin hover 隔離徽章，**Then** 顯示觸發原因與具體數字（calls / baseline / 倍數）
2. **Given** admin 開解除隔離頁，**When** 頁面呈現，**Then** 顯示該次隔離的具體觸發數據，不需查稽核紀錄
3. **Given** 一筆被手動暫停的分配，**When** admin 檢視，**Then** 顯示暫停（與隔離區分）的狀態說明

---

### 使用者故事 5 - 管理員按班級/群組看花費占比（優先級：P3）

管理員在首頁看到「Top 5 tags by spend」卡片——花最多的 5 個 tag（班級/群組/專案）。這延伸階段 15 的 tag 聚合資料到首頁的決策視角。

**為何此優先級**：階段 15 已交付 tag 聚合資料，首頁卡片是順手延伸；但它是錦上添花，非核心三圖。

**驗收情境**：

1. **Given** 成員掛了 tag 且有用量，**When** admin 看首頁，**Then** Top 5 tags by spend 卡片顯示花最多的 5 個 tag
2. **Given** 點一個 tag，**When** admin 點擊，**Then** 跳到用量頁的該 tag 視圖

---

### 邊界情況

- 平台完全沒有用量資料（全新部署）→ 每張圖顯示空狀態（友善 illustration / 文字），不是空白或報錯
- 某維度只有一個值（如只用一個 model）→ 環圈圖正常顯示單一區塊，不報錯
- 時段選擇器選了未來區間 / 起 > 訖 → 比照既有用量頁回驗證錯誤
- 時段過寬（如自訂 1 年）→ 沿用既有用量頁的區間上限規則
- 圖表資料量大（數萬筆呼叫）→ 渲染不卡頓（聚合在後端、前端只收聚合結果）
- 隔離分配無 details（舊資料缺觸發數據）→ 徽章 hover 顯示「原因未記錄」而非空白或報錯

## 需求 *(mandatory)*

### 功能需求

**圖表基建**

- **FR-001**: 系統 MUST 導入**單一**圖表呈現能力（整個平台統一一套，不混用多套）
- **FR-002**: 所有圖表的資料 MUST 在後端聚合後提供，前端只接收聚合結果（不在前端對原始呼叫資料做大量運算）
- **FR-003**: 系統 MUST 提供平台級（跨所有分配）的每日花費/用量時序資料（現有時序僅限單一分配）

**首頁圖表（≤ 3 張）**

- **FR-004**: 首頁 MUST 顯示過去 7–14 天的每日花費/用量長條圖，可切換 token 與花費兩種度量，hover 顯示當日明細
- **FR-005**: 首頁 MUST 顯示各 model 花費占比（top 5 + 其他），點一塊可跳該 model 詳情
- **FR-006**: 首頁 MUST 顯示花費最高的前 5 筆分配（友善名 + 數字），點一筆可跳該分配詳情
- **FR-007**: 首頁圖表總數 MUST ≤ 3 張（避免視覺超載）
- **FR-008**: 既有的隔離/暫停警示卡 MUST 維持在所有圖表**之上**、最顯眼處——圖表不得削弱或淹沒它

**用量頁進階視覺**

- **FR-009**: 用量頁 MUST 顯示 provider 花費/用量占比
- **FR-010**: 用量頁 MUST 顯示 24 小時 × 7 天的使用熱力圖，軸標清楚對非技術 admin 可讀

**時段選擇器**

- **FR-011**: 有圖表的頁面 MUST 提供統一時段選擇器（本週/本月/本季/自訂）
- **FR-012**: 切換時段 MUST 一次更新該頁所有圖表；更新中有載入指示
- **FR-013**: 自訂區間 MUST 沿用既有用量查詢的區間驗證與上限規則

**暫停/隔離原因顯眼化**

- **FR-014**: 分配列表的隔離/暫停徽章 MUST 可顯示觸發原因與具體數據（如「1 小時 1100 calls，baseline 100/hr」），不需點進稽核紀錄
- **FR-015**: 解除隔離頁 MUST 顯示該次隔離的具體觸發數據
- **FR-016**: 暫停（手動）與隔離（自動）的狀態說明 MUST 可區分
- **FR-017**: 隔離資料缺觸發細節時 MUST 顯示「原因未記錄」而非空白或報錯

**Tag 花費卡片**

- **FR-018**: 首頁 MUST 顯示「Top 5 tags by spend」卡片（延伸階段 15 的 tag 聚合），點 tag 跳用量頁該 tag 視圖

**通用品質**

- **FR-019**: 每張圖 MUST 對應一個會驅動決策的問題；無此性質的圖不納入
- **FR-020**: 圖表元素 MUST 可點擊跳轉到對應詳情頁（不要孤立圖）
- **FR-021**: 無資料時每張圖 MUST 顯示友善空狀態，不報錯不空白
- **FR-022**: 既有頁面與功能 MUST 零退化

### 主要實體

- **平台級時序聚合（查詢概念）**: 跨所有分配、按日（或時）聚合的花費/用量/呼叫數時間序列；無新持久化，查詢時聚合。
- **既有聚合維度**: member / allocation / model / tag（階段 15）— 圖表的資料源，沿用不改。
- **隔離/暫停觸發資料**: 既有稽核事件 `details`（含 last_hour_calls / baseline / reason）— 顯眼化的資料源，沿用不改。

## 成功標準 *(mandatory)*

### 可量測結果

- **SC-001**: 管理員能在首頁（不點進其他頁）回答「最近花錢節奏、錢燒在哪個 model、誰是重度使用者」三個問題
- **SC-002**: 首頁圖表數量 ≤ 3 張，且隔離/暫停警示在所有圖表之上、第一眼可見
- **SC-003**: 導入**恰好一個**圖表呈現方案；前端建置成功且套件體積增幅合理（gzip 增量 < 150KB）
- **SC-004**: 管理員能用時段選擇器把任一有圖頁面切到本週/本月/本季/自訂，所有圖表一起更新
- **SC-005**: 管理員能不點進稽核紀錄，就在分配列表或解除頁看到隔離/暫停的觸發原因與數據
- **SC-006**: 全新部署（無資料）時每張圖顯示友善空狀態，無報錯、無空白頁
- **SC-007**: 既有用量/首頁/分配功能零退化（既有測試全綠）

## 假設

- 圖表的資料源（member/allocation/model/tag 聚合、per-allocation 時序、稽核 details）多已存在；本階段主要新增「平台級時序」一個後端聚合，其餘為前端視覺化。
- 導入單一圖表方案（一個 lib 或其包裝）；不混用多套，避免套件體積膨脹與風格不一。
- 圖表聚合一律在後端完成，前端只收聚合結果——大用量下不卡頓。
- 「每張圖回答一個會驅動決策的問題」為硬性設計準則；不為「好看」加無決策價值的圖。
- 首頁最多 3 張圖；隔離/暫停警示永遠優先於圖表的視覺位階。
- Tag 卡片延伸階段 15 已交付的 tag 聚合資料；本階段不改 tag 聚合邏輯。
- 不做 allocation/member 詳情頁的圖（留未來增益）；不做花俏圖型（3D pie / radar / treemap）。
- 圖表更新頻率以既有用量資料的即時性為準（呼叫即記、查詢即聚合），無另設快取層；避免「美但 stale」。
