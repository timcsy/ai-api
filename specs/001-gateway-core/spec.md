# Feature Specification: 階段 1 — 分流核心 (Gateway Core MVP)

**Feature Branch**: `001-gateway-core`
**Created**: 2026-05-21
**Status**: Draft
**Input**: User description: "階段 1 分流核心"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 建立分配並成功代理呼叫 (Priority: P1)

組織內的擁有者（管理員）需要為一位開發者建立一筆 AI API 的「分配」。
建立後系統發行一張獨立憑證，開發者拿著這張憑證透過閘道呼叫 Azure OpenAI，
得到正確回應。整個過程中，開發者**不會接觸到**底層 Azure OpenAI 的 key。

**Why this priority**: 這是專案的存在意義最小切片——能分配資源、能用、能保護
底層憑證。少了它，其他功能都沒意義。

**Independent Test**: 由管理員介面／API 建立一筆綁定到「指定 Azure OpenAI
模型」的分配，取得憑證；用該憑證對閘道發出聊天補全請求，驗證回應內容來自
Azure OpenAI、且回應與錯誤訊息中**從未出現**底層 Azure OpenAI key 字串。

**Acceptance Scenarios**:

1. **Given** 系統內已設定 Azure OpenAI 端點與 key（管理員設定，非使用者可見），
   **When** 管理員建立一筆分配並指定模型，**Then** 系統回傳一張獨立憑證
   （與底層 Azure OpenAI key 不同）。
2. **Given** 持有上一步發行的憑證，**When** 以該憑證對閘道發出對應模型的呼叫，
   **Then** 閘道將請求代理至 Azure OpenAI 並回傳結果。
3. **Given** 任意呼叫情境（含錯誤情境），**When** 觀察回應、HTTP header、
   錯誤訊息、日誌，**Then** 任何位置都不得包含底層 Azure OpenAI key 的明文。

---

### User Story 2 - 撤回分配，後續呼叫即刻遭拒 (Priority: P1)

擁有者必須能在任何時刻撤回一筆已發行的分配；撤回之後，該憑證的後續呼叫
必須**立即**被拒絕，不依賴憑證自然過期。

**Why this priority**: 「分享是可撤回的授權」是根公理的直接推論——失去這個
能力，整個信任模型崩潰。與 P1 同列為必要 MVP。

**Independent Test**: 對一張仍有效的憑證連續打三次呼叫——前兩次成功；在第二
與第三次之間呼叫撤回 API；觀察第三次（在預設 SLO 內發出）必須收到拒絕。

**Acceptance Scenarios**:

1. **Given** 一張有效憑證，**When** 擁有者呼叫撤回 API，**Then** 系統回應撤回
   成功並標記該分配為 `revoked`。
2. **Given** 上一步剛被撤回的憑證，**When** 在撤回完成後的 SLO 內（≤ 5 秒）
   發出新呼叫，**Then** 閘道拒絕該呼叫並回應對應錯誤碼。
3. **Given** 同一擁有者其他分配仍有效，**When** 撤回一筆分配，**Then** 其他
   分配不受影響。

---

### User Story 3 - 呼叫可追溯到分配 (Priority: P1)

每一次成功或被拒絕的呼叫，必須能在事後由稽核者根據憑證或分配 ID 找到對應
的分配記錄、被分配對象、所綁定的資源。

**Why this priority**: 對應原則 2「可追蹤性」。對組織內部分享來說，「誰用了
什麼、用了多少」是擁有者最基本的知情權。

**Independent Test**: 以兩張不同分配的憑證各發出若干次呼叫；事後查詢呼叫
日誌應能：(a) 過濾出單一分配的所有呼叫；(b) 由任一筆日誌行回查到該分配的
被分配對象與綁定資源。

**Acceptance Scenarios**:

1. **Given** 多筆已發行的分配各自產生呼叫，**When** 由分配 ID 查詢日誌，
   **Then** 系統回傳該分配的所有呼叫（含時間、模型、token 數、結果）。
2. **Given** 一筆呼叫日誌，**When** 從中取出分配 ID 查詢分配記錄，**Then**
   可取得被分配對象識別與資源綁定欄位。
3. **Given** 不帶憑證或帶無效憑證的請求，**When** 閘道處理，**Then** 拒絕
   並留下匿名拒絕紀錄；**不得**將該請求歸屬到任何分配。

### User Story 4 - 宣告式部署與安全更新 (Priority: P2)

維運者要能以宣告式方式把閘道部署到 K8s，並且 LiteLLM 鏡像版本能自動追蹤
上游、有清楚的回滾路徑，確保安全性修補不滯後。

**Why this priority**: 願景架構的明確要求。雖然不在「單筆呼叫」核心路徑上，
但「安全性修補不滯後」是組織內服務上線的前提；不做這一塊，MVP 等於不能
進入生產。

**Independent Test**: 從乾淨叢集套用 manifests/Helm chart → 服務上線 →
觸發鏡像版本更新流程 → 故意提供一個失敗版本，驗證可在幾分鐘內回到上一版
且服務恢復可用。

**Acceptance Scenarios**:

1. **Given** 一份乾淨 K8s 開發叢集，**When** 套用本專案的部署描述
   （Helm 或 Kustomize），**Then** 閘道服務上線且健康檢查通過。
2. **Given** LiteLLM 上游有新版鏡像，**When** 自動更新流程觸發，**Then**
   閘道升級到新版且保留前一版引用。
3. **Given** 新版升級後出現異常（健康檢查失敗或人工判定），**When** 維運者
   觸發回滾，**Then** 服務在 ≤ 5 分鐘內回到前一版可用狀態。

---

### Edge Cases

- 撤回一筆已撤回的分配：應為冪等，回傳「已撤回」狀態，不視為錯誤。
- 同一憑證在撤回瞬間正在進行的請求：實作必須選擇明確策略（例：完成當前請求
  但拒絕後續），且該策略需被記錄於日誌。
- 分配所綁定的模型在 Azure OpenAI 端不存在或未部署：閘道回應 4xx，且錯誤
  訊息不得包含底層端點細節。
- Azure OpenAI 暫時不可用：閘道回應 5xx 並記錄歸屬於該分配的失敗呼叫；不得
  讓使用者誤以為是自身憑證問題。
- 同時兩位管理員撤回同一分配：以最早成功的撤回為準，第二次為冪等成功。
- 系統重啟期間發出的呼叫：撤回後的拒絕狀態必須持久化，不能因重啟而失效。

## Requirements *(mandatory)*

### Functional Requirements

#### 分配生命週期
- **FR-001**: 系統 MUST 提供建立分配的能力，輸入至少包含「被分配對象識別」
  與「資源綁定」（第一階段資源以 Azure OpenAI 模型名稱為粒度）。
- **FR-002**: 系統 MUST 為每一筆分配發行一張全域唯一、隨機產生、與底層
  Azure OpenAI key 不同的憑證。
- **FR-003**: 系統 MUST 提供撤回分配的能力，撤回為冪等操作。
- **FR-004**: 系統 MUST 將分配狀態（active / revoked）持久化，並在程序重啟
  後保持。

#### 代理呼叫
- **FR-005**: 閘道 MUST 接受以發行憑證呈交（標準 Authorization header）的呼叫
  並將之代理至對應的 Azure OpenAI 模型。
- **FR-006**: 閘道 MUST 在每次呼叫時**驗證分配的當前狀態**（非僅憑證簽章），
  狀態為 `revoked` 即拒絕。
- **FR-007**: 系統 MUST 確保自憑證撤回起至呼叫遭拒生效的時間 ≤ 5 秒（SLO）。
- **FR-008**: 閘道 MUST 拒絕使用「未綁定於該分配之模型」的呼叫，並以 4xx
  回應。

#### 機密保護
- **FR-009**: 底層 Azure OpenAI key MUST 僅存在於服務端設定（環境變數／
  Secret），**禁止**出現在 API 回應、錯誤訊息、HTTP header、日誌、追蹤資料。
- **FR-010**: 系統 MUST 透過契約測試確認 FR-009：對所有對外端點與錯誤路徑
  掃描回應內容，不得匹配到底層 key 的明文。

#### 可追蹤性
- **FR-011**: 系統 MUST 在每一次呼叫紀錄中包含分配 ID、被分配對象、所用
  模型、token 用量、結果碼、時間戳。
- **FR-012**: 系統 MUST 提供以「分配 ID」查詢該分配所有呼叫紀錄的能力。
- **FR-013**: 系統 MUST 拒絕並記錄無分配 ID 可歸屬的請求為「匿名拒絕」，
  不可歸屬到任何分配。

#### 契約與測試（依據 constitution）
- **FR-014**: 對外端點 MUST 先以 OpenAPI 契約定義，所有實作必須通過契約測試。
- **FR-015**: 所有 FR 必須先於實作存在對應的失敗測試（TDD），且測試在實作
  完成後轉為通過。

#### 部署與更新
- **FR-016**: 系統 MUST 提供宣告式 K8s 部署描述（Helm chart 或 Kustomize），
  支援單一指令安裝至開發叢集。
- **FR-017**: 系統 MUST 配置 LiteLLM 鏡像版本的自動化追蹤機制（例：Renovate
  /Dependabot），更新需經 CI 驗證才能套用至叢集。
- **FR-018**: 系統 MUST 提供至少一條回滾路徑，使任一次失敗更新可在 ≤ 5
  分鐘內回到前一可用版本。

#### 不在本階段範圍
- **FR-019** (NON-GOAL): 階段 1 不實作 Google Workspace SSO 或彈性認證
  ——管理員 API 以 bootstrap admin token（環境變數注入）保護即可。
- **FR-020** (NON-GOAL): 階段 1 不實作配額、過期、轉分配、用量觀測 UI、
  費用計算。

### Key Entities

- **Allocation（分配）**：代表一次「資源被分配給對象」的事件。
  - 屬性：分配 ID、被分配對象識別、資源綁定（模型名稱）、狀態（active /
    revoked）、建立時間、撤回時間（可空）、發行的憑證指紋。
  - 關係：1 分配 ↔ 1 憑證；1 分配 ↔ N 呼叫紀錄。
- **Credential（憑證）**：對外發出的隨機 token，繫於單一分配。
  - 屬性：token 值（僅在建立時回傳明文，之後僅存指紋）、分配 ID 反向關聯。
- **CallRecord（呼叫紀錄）**：每一次經閘道的呼叫紀錄。
  - 屬性：分配 ID（或 null 表匿名拒絕）、被分配對象、模型、token 用量、
    結果碼、時間戳、請求 ID。
- **Subject（被分配對象）**：階段 1 為任意字串識別（例：email 或暱稱），
  尚未綁定真正的身份系統。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 自管理員建立分配起，被分配者可在 1 分鐘內以發行憑證成功呼叫
  Azure OpenAI 一次。
- **SC-002**: 撤回操作後 5 秒內，原憑證的呼叫遭拒比例為 100%。
- **SC-003**: 對 100 次隨機抽樣的呼叫紀錄與錯誤回應做正規表達式掃描，找到
  底層 Azure OpenAI key 字串的次數為 0。
- **SC-004**: 任意一筆呼叫紀錄都能在 5 秒內由分配 ID 反查到被分配對象與
  綁定資源。
- **SC-005**: 從乾淨開發叢集到服務上線（健康檢查通過）≤ 10 分鐘，且使用的
  指令數 ≤ 5。
- **SC-006**: 模擬一次失敗更新並執行回滾，服務恢復可用時間 ≤ 5 分鐘。
- **SC-007**: 所有對外端點 100% 有 OpenAPI 契約與通過的契約測試。
- **SC-008**: 所有 FR 對應的測試於 git 歷史中可看到「測試先於實作」（測試
  commit 早於對應實作 commit）。

## Assumptions

- 部署目標叢集已具備基本能力：Ingress / 對外存取、Secret 管理、持久化儲存。
  叢集營運（節點、網路、儲存）由組織既有 IT 流程負責，不在本階段範圍。
- Azure OpenAI 已由管理員預先佈署所需模型，本階段以單一 endpoint + key
  的設定形式注入；多端點負載與多供應商抽象延後階段處理。
- 第一階段管理員 API 的存取控制以單一 bootstrap admin token 保護，token
  以環境變數／Secret 注入，不在 spec 範圍內處理「管理員身份系統」。
- 「被分配對象」第一階段以任意字串識別（例 email 或暱稱）即可，正式身份
  整合於階段 2。
- 撤回生效 SLO 設定為 5 秒，是基於「人因操作可接受感」與「實作可達成性」
  的折衷；未來可在加入快取／多副本後重新評估。
- LiteLLM 為 OSS 上游選型，自動更新由 GitOps（例：Renovate + Argo CD 或
  Flux）實作；具體工具選型於 plan 階段決定。
