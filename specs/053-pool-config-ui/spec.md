# Feature Specification: 配額池設定移到前端（admin 可編輯 T／保底 + 建議值）

**Feature Branch**: `053-pool-config-ui`
**Created**: 2026-06-29
**Status**: Draft
**Input**: User description: "配額池總額 T 與保底從 Helm/env 移到 DB，admin 在配額池監控頁可編輯（含驗證），並依近月用量顯示建議 T／保底與原因、可一鍵套用"

## 背景與問題

自適應配額池（每月把總額 T 依用量再分配給各分配）的兩個關鍵旋鈕——**總額 T** 與**每分配保底**——目前是部署層（Helm／環境變數）的設定，調整必須由工程師改設定再重新部署；管理員也無從在介面得知「**該設多少**」。但 T／保底本質是**業務/治理決策（配額）**，不是基礎設施參數——管理員應該能自助調整，且系統應主動給出有根據的建議。現況違反「被分配者/管理者要能在不依賴工程師下自助管理」的可達性精神。

本功能把 T／保底搬到**管理員介面可編輯**（資料存於平台、單一真理），並在既有「配額池監控」頁就地顯示**建議值與原因**（依近月實際用量），一鍵套用——調配額不再需要重新部署，也不再靠工程師。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 管理員在介面設定 T／保底並即時生效（Priority: P1）

管理員在「配額池監控」頁直接填入總額 T 與保底並儲存；下一次再分配（每月自動或按「手動執行再分配」）即採用新值，**全程不需重新部署**。

**Why this priority**: 這是本功能的核心——把配額決策權從「工程師 + 重部署」交還給管理員自助。

**Independent Test**: 在監控頁設定一組新的 T／保底並儲存，按「手動執行再分配」，確認再分配以新值計算、各分配的月配額隨之更新。

**Acceptance Scenarios**:

1. **Given** 管理員在監控頁，**When** 填入 T 與保底並儲存，**Then** 設定被持久化、頁面顯示的「目前生效值」即為剛存的值，且**未經重新部署**。
2. **Given** 已存的 T／保底，**When** 按「手動執行再分配」，**Then** 再分配以該值計算、寫入再分配紀錄、各分配月配額更新。
3. **Given** 管理員儲存設定，**Then** 留下稽核紀錄（誰、何時、改成什麼）。

---

### User Story 2 - 介面就地給出「該設多少」的建議與原因（Priority: P1）

管理員在同一頁看到一塊**建議區**：依近月實際用量算出建議的 T 與保底，附簡短原因與必要約束，可一鍵套用到表單。

**Why this priority**: 不會算/不知道設多少，正是現況的痛點之一；建議值讓管理員不必自己撈數據估算。與 US1 同為本功能價值主體，故並列 P1。

**Independent Test**: 開監控頁，確認建議區顯示近月用量、建議 T、建議保底與原因；按「套用建議」後表單填入該值。

**Acceptance Scenarios**:

1. **Given** 平台已有近月用量，**When** 管理員看建議區，**Then** 顯示近月用量、建議 T、建議保底，並說明各自原因（例如「建議 T ≈ 近月用量的 2 倍以保留成長空間並設總量上限」「保底太低會讓上月零用量的人本月被卡住」）。
2. **Given** 建議值，**When** 管理員按「套用建議」，**Then** 表單帶入建議的 T 與保底（仍可手動微調後再儲存）。

---

### User Story 3 - 防呆驗證，避免設出會出事的值（Priority: P2）

管理員填值時，系統擋下/警告會造成問題的設定：低於數學下限的擋下、會開始擋用量的給警告。

**Why this priority**: 配額設錯的後果是「全池無法再分配」或「使用者被無預警擋下」，需要護欄。依賴 US1 的表單存在。

**Independent Test**: 在表單輸入各種值，確認 T 小於「保底 × 池內成員數」時無法儲存、T 低於近月用量時出現警告。

**Acceptance Scenarios**:

1. **Given** 池內成員數 N 與保底，**When** 管理員把 T 設成小於「保底 × N」，**Then** 儲存被擋下並說明原因（總額不足以墊付每人保底）。
2. **Given** 表單，**When** T 低於近月實際用量，**Then** 顯示警告（這會讓部分使用者本月被擋），但仍允許管理員確認後儲存。
3. **Given** 表單，**When** 輸入負數或非數字，**Then** 被擋下。
4. **Given** 監控頁，**Then** 顯示目前池內成員數 N，並標示「設定於下次再分配生效」。

---

### Edge Cases

- **尚未設定過（首次）**：在管理員第一次於介面儲存前，系統沿用部署層既有值作為**初始預設**，行為與現況一致（不因搬家而改變現狀）。
- **單一真理**：設定只有一個可改處（管理員介面）。部署層的值僅在「介面從未設定過」時作為初始預設；一旦介面設過，以介面值為準，避免兩處不一致。
- **停用池**：把 T 設為 0 仍代表停用池（與現況語意一致），介面應明確顯示「已停用」。
- **生效時機**：改了設定但還沒到下次再分配、也沒按手動鍵時，舊的每分配配額仍在用——介面要講清楚「下次再分配才生效」。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 管理員 MUST 能在「配額池監控」頁檢視並編輯**總額 T** 與**每分配保底**，儲存後持久化於平台（不需重新部署即生效於下次再分配）。
- **FR-002**: 再分配計算與監控頁顯示的「目前生效值」MUST 來自**同一個單一真理**（管理員設定的值）；不得出現「顯示值 ≠ 實際採用值」。
- **FR-003**: 系統 MUST 在管理員從未於介面設定過時，沿用部署層既有值作為**初始預設**（搬移不改變現狀行為）。
- **FR-004**: 系統 MUST 依**近月實際用量**計算並顯示建議的 T 與保底，附簡短原因；管理員 MUST 能一鍵把建議值套用到表單。
- **FR-005**: 系統 MUST 擋下 **T < 保底 × 池內成員數 N** 的儲存，並說明原因。
- **FR-006**: 系統 MUST 在 **T 低於近月用量**時對管理員警告（仍可確認後儲存）；MUST 擋下負數/非法輸入。
- **FR-007**: 監控頁 MUST 顯示目前**池內成員數 N**，並標示設定「**於下次再分配生效**」。
- **FR-008**: 儲存設定 MUST 留稽核紀錄（操作者、時間、新值）。
- **FR-009**: T = 0 MUST 仍表示停用池（與現況語意一致），介面明確顯示停用狀態。

### Key Entities

- **配額池設定（Pool Config）**：平台層的單例設定，含「每月總額 T」與「每分配保底」兩個值 + 最後更新者/時間。為再分配與監控頁的單一真理；未設定時回退至部署層初始預設。
- **配額建議（Suggestion）**：依近月用量導出的唯讀計算結果（建議 T、建議保底、原因、約束所需的池內成員數 N）；非持久化，每次查詢即算。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理員在介面改 T／保底後按「手動執行再分配」，各分配月配額即以新值更新——**全程零重新部署**。
- **SC-002**: 監控頁顯示的「目前生效值」與再分配實際採用的值**永遠一致**（同一來源）。
- **SC-003**: 建議區依近月用量給出建議 T／保底 + 原因，一鍵可套用。
- **SC-004**: T < 保底×N 無法儲存；T < 近月用量時有警告；負數/非法輸入被擋。
- **SC-005**: 首次（介面未設定過）行為與現況一致——沿用部署層既有值，不需任何介面操作即維持原狀。
- **SC-006**: 設定變更可在稽核紀錄追溯。

## Assumptions

- **單一真理在平台資料層**：管理員設定為唯一可改處；部署層的 T／保底退為「介面未設定過時」的初始預設（不保留第二個可編輯路徑，避免漂移）。沿用既有「平台層單例設定 + 管理員介面自助」的前例（通知設定）。
- **建議公式**（informed default，可於介面文字說明）：建議 T ≈ 近月用量 × 2（保留成長空間且設總量上限）；建議保底 ≈ 讓上月零用量成員仍有可用基本額的量級；硬約束 T ≥ 保底 × N。實際係數可由管理員微調。
- **生效時機**：設定於下次再分配（每月排程或手動鍵）生效，非即時改寫既有配額——與現有再分配機制一致。
- **範圍邊界**：本功能只搬移/介面化 T／保底兩個旋鈕並加建議與驗證；**不改再分配演算法本身**、不動每分配手動配額管理、不做每日上限。
- 對應願景**階段 39**。
