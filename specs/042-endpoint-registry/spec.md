# Feature Specification: 統一端點架構（資料驅動 registry）+ moderation / search / image_edit 端點

**Feature Branch**: `042-endpoint-registry`
**Created**: 2026-06-11
**Status**: Draft
**Input**: User description: "統一端點架構 registry + moderation/search/image_edit 端點"

## User Scenarios & Testing *(mandatory)*

本功能把目前一個個手寫、結構幾乎複製的成員推論端點，收斂成**一條共用執行流程 + 一份端點註冊表（資料）**，讓「新增一個同形態端點」變成「加一筆資料」而非「複製一個檔」；並以此架構**按需補上三個同步推論端點**（內容審核 moderation、網路搜尋 search、圖片編輯 image_edit），驗證架構的可擴展性。

### User Story 1 - 既有端點零回歸地收斂到統一架構（Priority: P1）🎯 MVP

維運者（開發/管理）希望「新增或維護一個推論端點」不再是複製一份近乎相同的程式、而是宣告一筆描述其差異的資料。本故事把現有的成員推論端點（對話、responses、向量、文件辨識、圖片生成、重排序、語音合成、語音轉文字）統一到同一條執行流程，每個端點只保留「**它與其他端點真正不同的部分**」（如何收請求、用什麼單位計量、呼叫哪個上游、必填哪些欄位）。對成員而言**行為完全不變**——這是內部架構整併，外部契約與計費結果零變動。

**Why this priority**: 它是後續所有端點擴展的地基；沒有它，每加一個端點就多一份複製品、技術債持續累積。它本身即可獨立交付（純重構、零行為變更、以既有測試為金鋼罩），是這個主題的 MVP。

**Independent Test**: 重構後，現有每一個推論端點的行為、回應、計量計費結果與重構前**逐一相同**（既有契約/整合測試全數通過、無修改斷言）；且新增一個假端點只需註冊一筆資料、不需新增執行流程程式。

**Acceptance Scenarios**:

1. **Given** 重構完成，**When** 成員呼叫任一既有端點（對話/向量/OCR/圖片/rerank/TTS/STT/responses），**Then** 回應形態、計量單位、計費金額、歸戶分配與重構前完全一致。
2. **Given** 任一既有端點，**When** 觸發拒絕路徑（壞金鑰/未授權模型/缺欄位/上游錯誤），**Then** 錯誤碼、狀態碼、用量記錄與重構前一致。
3. **Given** 統一架構，**When** 要加一個與既有同形態（如純 JSON）的新端點，**Then** 只需新增一筆端點描述資料（路徑、上游函式、必填欄位、計量規則），不需改動共用執行流程。
4. **Given** 重構，**When** 跑既有測試套件，**Then** 全數通過且**未修改既有測試斷言**（行為不變的證明）。

### User Story 2 - 成員呼叫內容審核（moderation）模型（Priority: P2）

成員送一段文字，取回該文字的內容安全分類（是否含違規內容），用量以 token 計量計費、歸戶分配。

**Why this priority**: moderation 是最單純的新端點（純 JSON 進出、token 計費），用來**第一個驗證**「加端點＝加一筆資料」——若它能只靠註冊一筆資料就上線，架構即成立。

**Independent Test**: 有效金鑰 + moderation 模型 → 送文字 → 取回分類結果 + 記一筆 token 計費歸戶；該端點的程式增量僅為一筆註冊資料。

**Acceptance Scenarios**:

1. **Given** 一個 moderation 模型分配 + 金鑰，**When** 成員送文字，**Then** 取回安全分類、記一筆成功用量（token 計量、按價計費、歸戶分配）。
2. **Given** 缺輸入文字，**When** 呼叫，**Then** 回請求格式錯誤。
3. **Given** 壞/缺金鑰或範圍外模型，**When** 呼叫，**Then** 被擋下。
4. **Given** 上游失敗，**When** 呼叫，**Then** 回帶原因錯誤、記上游錯誤用量。

---

### User Story 3 - 成員呼叫網路搜尋（search）模型（Priority: P2）

成員送一個查詢，取回網路搜尋結果，用量以「**每次查詢**」計量計費、歸戶分配。

**Why this priority**: search 雖也是 JSON，但其上游呼叫的**參數形狀與其他端點不同**（以「搜尋供應商 + 查詢」而非單純「模型 + 輸入」呼叫）——用來驗證註冊表能描述「每端點各異的請求→上游參數對映」，而非假設所有端點長一樣。

**Independent Test**: 有效金鑰 + search 模型 → 送查詢 → 取回搜尋結果 + 記一筆「每查詢」計費歸戶。

**Acceptance Scenarios**:

1. **Given** 一個 search 模型分配 + 金鑰，**When** 成員送查詢，**Then** 取回搜尋結果、記一筆成功用量（單位＝查詢、數量＝1、按每查詢價計費、歸戶分配）。
2. **Given** 缺查詢，**When** 呼叫，**Then** 回請求格式錯誤。
3. **Given** 未定每查詢價，**When** 呼叫，**Then** 回結果、記用量、花費以 0 計。
4. **Given** 上游失敗，**When** 呼叫，**Then** 回帶原因錯誤、記上游錯誤用量。

---

### User Story 4 - 成員呼叫圖片編輯（image_edit）模型（Priority: P3）

成員**上傳一張圖片**（加上文字提示），取回編輯後的圖片，用量以「**每張圖**」計量計費、歸戶分配。

**Why this priority**: image_edit 帶入「multipart 圖片上傳 + 非 token 單位（每張圖）」的組合——驗證註冊表能複用既有的上傳形態處理、並搭配「每張圖」計量。形態最複雜，故優先序最低。

**Independent Test**: 有效金鑰 + image_edit 模型 → 上傳圖片 + 提示 → 取回編輯後圖片 + 記一筆「每張圖」計費歸戶。

**Acceptance Scenarios**:

1. **Given** 一個 image_edit 模型分配 + 金鑰，**When** 成員上傳圖片 + 提示，**Then** 取回編輯後圖片、記一筆成功用量（單位＝圖、數量＝產出圖數、按每張圖價計費、歸戶分配）。
2. **Given** 缺圖片，**When** 呼叫，**Then** 回請求格式錯誤。
3. **Given** 未定每張圖價，**When** 呼叫，**Then** 回結果、記用量、花費以 0 計。
4. **Given** 上游失敗，**When** 呼叫，**Then** 回帶原因錯誤、記上游錯誤用量。

### Edge Cases

- **零回歸是硬約束**：重構後既有端點任何外部可觀察行為（回應、計量、計費、錯誤、歸戶）不得有差異。
- **每端點的差異要可宣告**：請求解析（JSON body / multipart）、上游參數對映（model+input vs search_provider+query vs image+prompt）、計量單位、必填欄位——皆以資料描述，不寫死在共用流程。
- **未定價（任一單位）**：回結果、記數量、花費以 0 計、不阻擋。
- **未支援的 mode**：目錄收進但本功能未涵蓋的 mode（video/realtime/vector_store 等）→ 類型判為「未知」、成員端不誤開（誠實機制延續）。
- **計量來源缺漏**：取不到計量數量時記數量為空、花費 0、仍記一筆（不無聲）。
- **上傳檔案**：image_edit 圖片上傳受既有請求大小上限約束。

## Requirements *(mandatory)*

### Functional Requirements

#### 統一架構（US1）

- **FR-001**: 系統 MUST 以**單一共用執行流程**處理所有成員推論端點：請求解析 → 既有前置檢查（金鑰/分配/存取/憑證）→ 上游呼叫 → 計量 → 計費歸戶 → 記錄 → 回應，以及統一的錯誤處理。
- **FR-002**: 每個端點的**差異 MUST 以資料描述**（至少含：對外路徑、上游函式、請求→上游參數的對映、必填欄位、計量規則、回應形態），不得把端點專屬邏輯散落在共用流程中。
- **FR-003**: 計量 MUST 可插拔：以 token 計（讀回應 usage）或以非 token 單位計（page/query/character/second/image…，依端點宣告）。
- **FR-004**: I/O 形態 MUST 以**少數可重用的形態處理器**承載（JSON 進出、binary 輸出、multipart 上傳…），新增同形態端點 MUST NOT 需要新增形態處理器。
- **FR-005**: 重構 MUST 對既有所有推論端點維持**完全相同的外部行為**（回應、計量、計費、錯誤、歸戶）——既有測試 MUST 全數通過且**不修改其斷言**。

#### 新端點（US2–US4）

- **FR-006**: 系統 MUST 提供內容審核端點，讓成員送文字、取回安全分類；以 token 計量計費。
- **FR-007**: 系統 MUST 提供網路搜尋端點，讓成員送查詢、取回搜尋結果；以「每查詢」計量計費；其上游參數對映以資料宣告（不假設與其他端點相同）。
- **FR-008**: 系統 MUST 提供圖片編輯端點，讓成員上傳圖片 + 提示、取回編輯後圖片；以「每張圖」計量計費。
- **FR-009**: 三個新端點的**程式增量 MUST 主要為註冊資料 + 必要的計量/形態複用**，不需各自重寫一條執行流程（架構成立的證明）。

#### 共通

- **FR-010**: 三個新端點 MUST 走與既有相同的前置檢查、未定價→0、上游錯誤可診斷且憑證去敏。
- **FR-011**: 本功能 MUST NOT 需要資料庫結構變更（計量單位為既有字串維度；新單位 image/query 為字串值）。
- **FR-012**: 未涵蓋的 mode（video/realtime/vector_store…）MUST 維持「類型未知、成員端不開」的誠實呈現。

### Key Entities *(include if feature involves data)*

- **端點描述（EndpointSpec）**：一筆描述某端點差異的資料——對外路徑、上游函式、請求→上游參數對映、必填欄位、計量規則、I/O 形態。新增同形態端點＝新增一筆。
- **計量策略（Meter）**：決定一次呼叫的計量方式——以 token（讀 usage）或以某非 token 單位（含數量計算）。與 I/O 形態正交。
- **I/O 形態處理器（IO Shape）**：少數可重用的請求收取/回應產出方式（JSON、binary 輸出、multipart 上傳）。
- **用量紀錄 / 價目 / 分配**：沿用既有（計量數量 + 單位、每單位價、歸戶對象），無結構變更。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 重構後，現有所有推論端點的外部行為（回應/計量/計費/錯誤/歸戶）與重構前**逐一相同**，既有測試全數通過且未修改斷言。
- **SC-002**: 新增一個同形態（JSON）端點所需的程式增量，**主要是一筆註冊資料**（可量化：不需新增共用執行流程或形態處理器）。
- **SC-003**: 成員能呼叫 moderation（token）、search（每查詢）、image_edit（每張圖）三個新端點，各取回正確結果並記一筆正確計量計費歸戶的用量。
- **SC-004**: search 的「上游參數與其他端點不同」與 image_edit 的「multipart 上傳 + 每張圖」皆由架構正確承載，無需為其特例化共用流程。
- **SC-005**: 資料庫維持單一 migration head；無新增第三方套件（multipart 上傳沿用既有能力）。
- **SC-006**: 未涵蓋的 mode 在目錄中仍誠實顯示為「未知類型」、不對成員開放。

## Assumptions

- **沿用既有前置檢查與計費一般化**：統一流程重用既有 endpoint-agnostic preflight 與「數量 + 單位」計費維度；新單位 image/query 為字串值。
- **三軸正交**：I/O 形態、計量策略、上游函式/參數對映彼此獨立，是架構的設計主軸。
- **本功能僅含同步推論端點**：非同步（video_generation 的 job 生命週期）、WebSocket（realtime）、資源管理（vector_store）**明確不在範圍**，留待各自獨立評估（多半 descope 或另立子系統）。
- **image_edit 計量為「產出圖數」**：對齊上游「每張圖」計價；圖片以上傳（multipart）輸入、編輯後圖片以資料形式回傳。
- **search 以「每查詢 1 單位」計量**：對齊上游「每查詢」計價；其上游以「搜尋供應商 + 查詢」呼叫，由端點描述資料對映。
- **不在本功能範圍**：每單位用量上限、圖表/視覺化改版、async/ws/vector_store 端點。
