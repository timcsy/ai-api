# Feature Specification: OpenAI 相容 `/v1/models` ＋ Copilot 上卡

**Feature Branch**: `050-openai-models-copilot`
**Created**: 2026-06-27
**Status**: Draft
**Input**: User description: "OpenAI 相容 /v1/models 端點（回呼叫金鑰 scope 內的模型）並驗證後新增 Copilot 應用卡"

## 背景與問題

平台對外是 OpenAI 相容 gateway，願景明言「**任何會講 OpenAI API 的客戶端都能指過來**」。但目前缺一個多數 OpenAI 客戶端在連線時會用到的**模型發現**端點：`GET /v1/models`。客戶端（GitHub Copilot、Continue、OpenWebUI、LibreChat、官方 SDK 的 `client.models.list()`…）指向本平台時，多會先打 `GET /v1/models` 拉可用模型清單；本平台目前在 `/v1` 下只提供推論端點（chat / responses / realtime / embeddings / ocr / images / rerank / audio / moderation / search / image_edit），打 `GET /v1/models` 會 404。平台自有的模型目錄在 `/catalog` 前綴、且非 OpenAI 格式，客戶端讀不到。

後果：使用者拿到金鑰、把 base URL 指過來，客戶端卻「列不出模型」或顯示連線異常——**名義可達、實質接不上**（原則 6 可達性的反面）。這也正是上一階段（階段 34）把「Copilot 上卡」延後的具體原因：Copilot 會打「模型清單 / embeddings」，要先確認 gateway 能日常支撐才正式上卡。

本功能補上 OpenAI 相容的模型發現端點，並以它為前提，真機驗證 GitHub Copilot 後把 Copilot 加為應用商店第二張「工具整合」卡。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 客戶端用金鑰列出可用模型（Priority: P1）

開發者（或任何 OpenAI 相容客戶端）把 base URL 指向本平台、填入分配到的應用金鑰後，呼叫「列出模型」會得到**這把金鑰實際能用的模型清單**（不多、不少），每個模型的識別碼正是後續呼叫時要填的 `model` 值。

**Why this priority**: 這是所有 OpenAI 相容客戶端連線的共同第一步，缺了它「指過來就能用」這個願景核心無法成立。它**獨立於 Copilot**——任何客戶端、甚至 `curl` 都立即受惠，是本功能的 MVP。

**Independent Test**: 帶一把有效金鑰對「列出模型」端點發一次請求，斷言回傳清單恰等於該金鑰 scope 內分配的模型；換一把 scope 不同的金鑰得到不同清單；無金鑰被拒。完全不需要 Copilot 即可驗證並交付價值。

**Acceptance Scenarios**:

1. **Given** 一把金鑰其 scope 含分配 A（模型 `azure/gpt-5.4`）與分配 B（模型 `azure/text-embedding-3-large`），**When** 客戶端用該金鑰列出模型，**Then** 回傳清單恰含這兩個模型的識別碼、且為 OpenAI 相容的清單結構（含每個項目的 `id` 與標示其為「模型」的型別欄位）。
2. **Given** 同一把金鑰，**When** 客戶端取回單一模型（指定其識別碼），**Then** 回傳該模型一筆、識別碼一致；指定不在 scope 內的識別碼則回「找不到」。
3. **Given** 一個沒帶金鑰或帶無效金鑰的請求，**When** 呼叫列出模型，**Then** 回傳未授權錯誤（與其他 `/v1` 端點一致），不洩漏任何模型資訊。
4. **Given** 一把金鑰，**When** 用它列出模型後，把清單裡的某個識別碼原樣當作 `model` 去呼叫對應推論端點，**Then** 該呼叫被正確路由、不發生「模型不在範圍」或識別碼對不上的錯誤。
5. **Given** 一把金鑰其 scope 內某模型尚未定價（無有效價目），**When** 列出模型，**Then** 該模型仍出現在清單（可被呼叫與否依既有存取政策決定），模型發現不因定價缺失而隱藏它。

### User Story 2 - 成員從應用商店接上 GitHub Copilot（Priority: P2）

成員在「應用」商店看到一張 **Copilot** 卡，照著卡上的步驟（設定連線 + 用捷徑建一把適合的金鑰）把 VS Code 的 GitHub Copilot 指向本平台，並能日常使用（列模型、對話）。

**Why this priority**: 直接回應真實使用者回報的接入痛點，延續階段 27/28 應用商店「拿到鑰匙後真的接得上工具」。依賴 US1（Copilot 會先打模型清單），故排 P2。

**Independent Test**: 在真實 VS Code + GitHub Copilot 對本平台跑一輪（列模型 → 一次對話）；應用商店出現 Copilot 卡、卡上步驟可照做。卡只在真機驗證「能日常用」後才正式上架（願景「確認能日常用再上卡」）。

**Acceptance Scenarios**:

1. **Given** 平台已支援模型發現（US1）且成員有至少一個相容分配，**When** 成員開「應用」商店，**Then** 看到一張 Copilot 卡（真實名稱 + logo + 一句話），點進有設定步驟與「建金鑰」捷徑。
2. **Given** 成員照 Copilot 卡步驟設定完成，**When** 在 VS Code 用 Copilot 對本平台發起一次對話，**Then** 對話成功完成、用量歸戶到對應分配。
3. **Given** 成員沒有任何 Copilot 可用的相容分配，**When** 看 Copilot 卡，**Then** 卡顯示「你目前沒有可用於此的分配」的指引（不給死路的建立鈕），與既有 Codex 卡的零分配處理一致。
4. **Given** 真機驗證發現 Copilot 某項功能在本平台無法可靠日常使用，**When** 上卡，**Then** 卡誠實標示該限制（或該功能延後），不列出「列了卻處處紅字」的步驟。

### User Story 3 - 跨 model／過期續接時得到可操作的指引（Priority: P3）

成員在同一把金鑰下切換 model（＝切換分配）或續用已過期的對話時，客戶端（如 Copilot 的伺服器端對話狀態模式）若嘗試接續舊對話而接不上，使用者會收到**清楚、可操作**的訊息（「請開新對話」），而非靜默地丟失上下文、或一句無資訊的紅字。

**Why this priority**: 後端「明確拒絕、不靜默降級」的抉擇已定案（見經驗教訓），本故事的增量是把錯誤訊息做得**可操作**並在 Copilot 卡上事先說明，降低使用者踩坑。屬體驗收尾，排 P3。

**Independent Test**: 用同一把金鑰、跨兩個分配（model）續用同一伺服器端對話，斷言回傳的是可操作的明確錯誤（指出「換 model／過期 ＝ 開新對話」），且 Copilot 卡文案已載明此行為。

**Acceptance Scenarios**:

1. **Given** 一個伺服器端對話建立於分配 A，**When** 客戶端在分配 B（同金鑰、不同 model）下嘗試以該對話識別碼續接，**Then** 回傳明確且可操作的錯誤訊息，內容讓使用者知道要開新對話，且不會把舊對話的上下文無聲帶到新分配。
2. **Given** 一個已過期、清理掉的對話，**When** 客戶端嘗試續接，**Then** 回傳明確的「對話不存在 / 請開新對話」訊息，而非靜默當成新一輪重開。
3. **Given** Copilot 卡，**When** 成員閱讀，**Then** 卡上已說明「同一把金鑰跨 model 切換＝開新對話」這個行為與原因（伺服器端記憶是 per-分配）。

### Edge Cases

- **空 scope 金鑰**：金鑰沒有任何有效分配時，列出模型回**空清單**（合法的空 list），非錯誤。
- **被停用/撤回的分配**：金鑰 scope 內某分配已暫停或撤回時，該模型**不應**出現在可用清單（與呼叫端的執法一致，避免「列得出卻打不通」）。
- **識別碼形式**：模型識別碼以何種形式回傳（含 provider 前綴與否）必須與「拿回來當 `model` 再呼叫」能對得上，避免重演 client picker 看不到 / 對不上的坑。
- **同名模型跨 provider**：同一把金鑰 scope 同時含 `azure/X` 與 `openai/X` 時，兩者都各自列出、識別碼可區分，不合併、不致歧義。
- **未定價模型**：出現在清單但不因定價缺失被隱藏；是否可呼叫仍由既有存取政策決定。
- **Copilot 的非對話請求**：Copilot 可能對 embeddings / 模型清單以外的端點發探測請求；驗證時需確認這些不會造成阻斷性錯誤，或卡上誠實標示限制。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 提供一個 OpenAI 相容的「列出模型」能力，讓帶有效應用金鑰的客戶端取得可用模型清單，回傳結構符合 OpenAI `GET /v1/models` 慣例（一個清單容器，內含多筆模型項目，每筆至少有識別碼與標示其為模型的型別欄位）。
- **FR-002**: 系統 MUST 把「列出模型」的範圍限定為**呼叫所用金鑰實際被授予的分配對應的模型**（金鑰 scope），不回傳該成員其他金鑰或其他分配的模型；語意為「這把金鑰能用哪些模型」。
- **FR-003**: 系統 MUST 提供 OpenAI 相容的「取回單一模型」能力（依識別碼），在 scope 內回該模型一筆、不在 scope 內回「找不到」。
- **FR-004**: 系統 MUST 對未帶金鑰或無效金鑰的模型發現請求回傳未授權錯誤，與其他對外推論端點的驗證行為一致，且錯誤不洩漏任何模型清單內容。
- **FR-005**: 「列出模型」回傳的模型識別碼 MUST 與客戶端後續呼叫推論端點時應填入的 `model` 值**完全一致**（拿回來原樣可用、能被正確路由到對應分配），不得出現識別碼形式不一致導致「找得到卻呼叫不到」。
- **FR-006**: 系統 MUST 在模型發現結果中排除「列得出卻打不通」的模型——金鑰 scope 內已暫停 / 撤回的分配對應模型不應出現在可用清單。
- **FR-007**: 模型發現 MUST NOT 因模型缺少定價而隱藏該模型（定價缺失不等於不可發現）。
- **FR-008**: 系統 MUST 對模型發現的每次請求沿用既有對外端點的前置驗證（身份 / 金鑰 / 存取政策），使列舉行為可追蹤、可歸屬（原則 2）。
- **FR-009**: 系統 MUST 在「應用」商店新增一張 **GitHub Copilot** 卡，內容包含：設定連線本平台的步驟、建立適合金鑰的捷徑、以及無可用相容分配時的指引（不給死路的建立鈕）。
- **FR-010**: Copilot 卡的內容 MUST 只反映**真機驗證為可日常使用**的設定；任一無法可靠驗證可用的功能，MUST 誠實標示限制或延後，不得列出已知會「處處紅字」的步驟。
- **FR-011**: 應用商店的新增 MUST 以既有註冊表機制承載（加一筆資料＝一張卡 + 一個詳情），不為單一應用改動商店架構（原則 7）。
- **FR-012**: 系統 MUST 在續接（伺服器端對話狀態）無法被忠實履行時（跨分配、過期）維持**明確拒絕、不靜默降級**，且錯誤訊息 MUST 可操作（讓使用者知道要開新對話）。
- **FR-013**: Copilot 卡 MUST 載明「同一把金鑰跨 model 切換＝開新對話」的行為與原因，使使用者事先知道此邊界。
- **FR-014**: 本功能 MUST NOT 改變既有任一推論端點的請求 / 回應與計費行為（純新增模型發現端點 + 前端卡 + 錯誤訊息文案優化）。

### Key Entities *(include if feature involves data)*

- **模型發現清單項目**：對外呈現「一個可用模型」的最小資訊——識別碼（＝呼叫時的 `model`）、標示其為模型的型別、（可選）擁有者 / 來源標示。資料來源為**既有模型目錄**經「金鑰 scope ∩ 存取政策」過濾，**不新增資料表、不新增持久化欄位**。
- **應用卡（Copilot）**：應用商店註冊表中的一筆——名稱、logo、一句話、設定步驟、建金鑰捷徑、限制說明。承載於既有前端註冊表，**非後端資料模型**。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 帶有效金鑰呼叫「列出模型」回傳的模型集合，與該金鑰 scope 內可呼叫的模型集合**100% 一致**（不多列無權限模型、不漏列可用模型）。
- **SC-002**: 從「列出模型」取得的任一識別碼，原樣當 `model` 呼叫對應端點，路由成功率 **100%**（無識別碼對不上導致的失敗）。
- **SC-003**: 一個 OpenAI 相容客戶端（含官方 SDK 的 `models.list()` 與 `curl`）指向本平台，**無需任何平台專屬調整**即可完成「列出模型」這一步。
- **SC-004**: 在真實 VS Code + GitHub Copilot 對本平台完成「列模型 → 一次對話」端到端流程**至少各一次成功**（真機驗收，非僅 401 煙霧）。
- **SC-005**: 成員從「應用」商店的 Copilot 卡出發、不需另一位 actor（工程師 / admin）協助，即可完成 Copilot 接入設定（對應原則 6 可達性）。
- **SC-006**: 跨分配 / 過期續接的情境下，使用者收到的訊息能讓其**正確採取下一步（開新對話）**，無「以為續接、實則失憶」的靜默上下文遺失。

## Assumptions

- **金鑰 scope 為模型發現範圍**（方案 b，沿用階段 34「如何使用**這把**金鑰」+ 原則 1 額度綁分配）：模型發現回傳的是呼叫金鑰被授予的分配對應模型，而非該成員全部可見目錄。若日後需要「列成員全部可用模型」另開語意，不在本功能。
- **模型識別碼採既有正規 slug（含 provider 前綴）**：與模型目錄與 proxy 路由所用識別一致，確保「列出 → 原樣呼叫」對得上；既有 bare-slug alias 機制（針對自帶 catalog 的 client）維持不變，本功能不依賴它。
- **重用既有模型目錄查詢服務**（credential gate ∩ access policy）作為資料源，**不新增資料表、不新增 migration、不新增第三方套件**。
- **Copilot 上卡以真機驗證為前提**：驗證在維護者的真實 VS Code 上進行；若驗證揭露阻斷性限制，卡得誠實標示或延後（SC-004 是驗收門檻，非自動化單測能涵蓋）。
- **續接 fail-loud 行為已存在**：US3 的後端拒絕邏輯（跨分配 forbidden / 過期 not-found）已上線，本功能只優化訊息可操作性與卡上說明，不改其「拒絕不降級」抉擇。
- **GitHub Copilot 的接入形態**以「VS Code Copilot 指向 OpenAI 相容自訂端點」為準；其他 IDE / Copilot 變體不在本功能範圍（可後續加卡或多介面）。
- 願景階段對位：本功能對應願景「任何會講 OpenAI API 的客戶端都能指過來」+ 階段 27/28 應用商店延伸，建議編為**階段 36**（與按需的階段 35 供應鏈並行、互不相依）。
