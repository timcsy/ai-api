# 功能規格：Tag-based 群組成本 rollup

**Feature Branch**: `023-tag-group-rollup`
**建立日期**: 2026-06-03
**狀態**: Draft
**輸入**: 使用者描述:「讓成本／用量可按 tag（班級／群組／專案）聚合，admin 不必自己出 Excel 加總。沿用既有 MemberTag 系統與 aggregate_usage 的 group_by 框架，新增 group_by=tag 維度。」

## 使用者情境與測試 *(必填)*

### 使用者故事 1 - 管理員按班級／群組檢視當月支出（優先級：P1）

管理員想知道「資訊一班這個月用了多少 AI 額度」。他打開用量頁、切到「依 Tag」視圖，看到一份依 tag 聚合的列表：每個 tag（如 `class-101`、`資訊社`、`project-x`）一列，顯示該 tag 底下所有成員在選定區間內的 token 總量、估算花費、呼叫次數。他不必再把同一班每個學生的數字手動抄進 Excel 加總。

**為何此優先級**：這是本功能的核心價值，也是外部回饋一致指出的學校／團隊剛需——預算是按班級給的、報告是按專案寫的，但平台原本只能按「個別成員」看，admin 得自己加總。消除這個手動加總是整個功能存在的理由。

**獨立可測**：seed 幾位掛了不同 tag 的成員 + 各自的呼叫紀錄，切到 Tag 視圖，驗證每個 tag 的聚合數字正確、且等於該 tag 成員各自數字之和。

**驗收情境**：

1. **Given** 有成員 A（tag `class-101`）用了 1000 tokens、成員 B（tag `class-101`）用了 500 tokens，**When** admin 以 tag 維度查當月用量，**Then** `class-101` 那列顯示 1500 tokens（兩人之和）
2. **Given** 選定一個時間區間，**When** admin 查 tag 用量，**Then** 只計入該區間內的呼叫，區間外的不計
3. **Given** 某 tag 底下成員在區間內完全沒有呼叫，**When** admin 查，**Then** 該 tag 顯示 0（或不出現），不會報錯
4. **Given** 平台有估算花費（已定價的 model），**When** admin 查 tag 用量，**Then** 每個 tag 顯示 token 與估算花費；遇未定價 model 比照既有用量頁標示「部分未定價」

---

### 使用者故事 2 - 管理員點開某 tag 看成員明細與重疊提示（優先級：P2）

管理員看到 `class-101` 這個月花得比預期多，想知道是誰。他點開 `class-101`，看到該 tag 底下每位成員的個別用量（token / 花費 / 次數）與加總。頁面同時明確標示：「成員可同時掛多個 tag，跨 tag 的加總可能重複計算」——因為一位學生若同時掛 `class-101` 與 `資訊社`，他的用量會同時計入兩個 tag，所以「所有 tag 的總和」不等於「平台總用量」。

**為何此優先級**：聚合數字若沒有「能下鑽到成員」就只是個數字，無法行動（admin 想處理時還是要知道是誰）。重疊提示則避免 admin 把「各 tag 加總」誤當成「組織總額」而誤判。

**獨立可測**：點開一個 tag，驗證列出的成員與其用量正確；且 UI 有重疊警示文字。

**驗收情境**：

1. **Given** `class-101` 有成員 A、B，**When** admin 點開 `class-101`，**Then** 列出 A、B 各自用量與加總
2. **Given** 成員 C 同時掛 `class-101` 與 `資訊社`，**When** admin 分別查兩個 tag，**Then** C 的用量在兩個 tag 都被計入（重疊），且 UI 明確標示此重疊性質
3. **Given** admin 在看 tag 明細，**When** 介面呈現，**Then** 有一句明確說明「跨 tag 加總可能重疊、不等於平台總額」

---

### 使用者故事 3 - 管理員匯出 tag 聚合供報告（優先級：P3）

管理員要把「各班這學期 AI 使用」放進給主任的報告。他在 Tag 視圖按匯出，得到一份 CSV / JSON，欄位為 tag、token、估算花費、呼叫次數、區間——可直接貼進試算表或文件。

**為何此優先級**：對外報告是學校場景的真實終點，但匯出是既有用量頁已有的能力（CSV/JSON），延伸到 tag 維度成本低；非阻擋核心價值（沒有匯出，admin 仍能在頁面上看到聚合）。

**獨立可測**：以 tag 維度呼叫匯出端點，驗證回傳格式正確、數字與頁面一致。

**驗收情境**：

1. **Given** Tag 視圖有資料，**When** admin 匯出 CSV，**Then** 每列一個 tag、欄位含 token/花費/次數，數字與頁面顯示一致
2. **Given** 同上，**When** admin 匯出 JSON，**Then** 結構與既有 member/allocation 維度匯出一致（只是 group_by 值為 tag）

---

### 邊界情況

- 沒有任何成員掛 tag → Tag 視圖顯示空狀態「尚無任何 tag 用量」，不報錯
- 一個 tag 底下成員全部沒有用量 → 該 tag 顯示 0 或不出現（與既有 member 維度「零用量成員」行為一致）
- 成員被停用 / 刪除但歷史呼叫仍在 → 聚合仍計入該歷史呼叫（用量是歷史事實，不因成員狀態改變）
- 成員的 tag 在區間內變動（中途被加 / 移除 tag）→ 採「查詢當下的 tag 歸屬」為準（不做 tag 的時間版本化，見假設）
- 區間參數非法（from > to）→ 比照既有用量頁回驗證錯誤
- 一位成員掛了大量 tag（如 10 個）→ 其用量在每個 tag 都計入，效能仍可接受（聚合走單一 JOIN）

## 需求 *(必填)*

### 功能需求

**聚合**

- **FR-001**: 系統 MUST 提供「依 tag 聚合用量」的能力，回每個 tag 在指定時間區間內的 token 總量、估算花費、呼叫次數
- **FR-002**: tag 的聚合 MUST 等於該 tag 底下所有成員各自用量之和（在同一區間、同一 filter 條件下）
- **FR-003**: 聚合 MUST 沿用既有用量維度（member/allocation/model）的時間區間語意與計費邏輯；遇未定價 model 比照既有標示
- **FR-004**: tag 的來源 MUST 為既有的成員-tag 歸屬資料；**不新增任何資料表或 schema 欄位**

**重疊語意**

- **FR-005**: 系統 MUST 正確處理「成員同時掛多 tag」——該成員用量計入其每一個 tag（刻意的重疊）
- **FR-006**: 介面 MUST 明確標示 tag 聚合的重疊性質：「成員可掛多 tag，各 tag 加總可能重複計算、不等於平台總用量」
- **FR-007**: 系統 MUST NOT 把「各 tag 用量加總」當成「平台總用量」呈現（總額仍以 member/allocation/model 維度為準）

**檢視與下鑽**

- **FR-008**: 管理員 MUST 能在用量頁切換到「依 Tag」視圖，看到各 tag 該區間的聚合
- **FR-009**: 管理員 MUST 能點開單一 tag，看到該 tag 底下每位成員的個別用量與加總
- **FR-010**: Tag 視圖 MUST 支援與既有用量頁相同的時間區間選擇

**匯出**

- **FR-011**: 系統 MUST 支援以 tag 維度匯出 CSV 與 JSON，欄位與數字與頁面一致

**權限**

- **FR-012**: Tag 聚合視圖 MUST 僅限管理員（admin）使用——它是治理／組織端視角，**不**出現在成員自助路徑（`/me`）
- **FR-013**: 一般成員 MUST NOT 能透過任何端點取得跨成員的 tag 聚合（避免成員看到別人的用量）

### 主要實體

- **Tag 用量聚合（檢視概念，非新資料表）**: 由「成員-tag 歸屬」與「呼叫紀錄」在查詢時 JOIN 聚合而成。粒度 = 一個 tag 一列，欄位：tag 名稱、token 總量、估算花費、呼叫次數、時間區間。無持久化、無新 schema。
- **Tag 成員明細（檢視概念）**: 某 tag 底下的成員清單，每位帶其個別用量；用於下鑽。

## 成功標準 *(必填)*

### 可量測結果

- **SC-001**: 管理員能在 30 秒內回答「某班／某群組這個月用了多少」——不需匯出原始資料自行加總
- **SC-002**: tag 聚合數字與「手動把該 tag 成員各自用量相加」100% 一致（可由測試逐筆驗證）
- **SC-003**: 跨 tag 重疊的成員，其用量正確地計入每一個所屬 tag，且介面讓 admin 不會把各 tag 加總誤判為平台總額
- **SC-004**: 本功能上線後，既有 member/allocation/model 維度用量查詢零退化（數字與行為不變）
- **SC-005**: 一般成員無法透過任何路徑取得跨成員的 tag 聚合（隔離可由測試驗證）
- **SC-006**: 不新增任何資料表、不新增 migration（純查詢層擴充）

## 假設

- 沿用既有「成員-tag 歸屬」資料作為唯一聚合來源；tag 名稱集合即既有資料的 distinct。不為本功能建任何新表或欄位。
- tag 歸屬**不做時間版本化**：聚合採「查詢當下成員的 tag」為準，不追溯「呼叫發生當時該成員掛哪些 tag」。理由：tag 是相對穩定的組織歸屬（班級／社團），歷史重歸屬罕見，時間版本化複雜度不符當前需求（YAGNI）。若未來有「學期中轉班需精確歷史歸屬」需求再評估。
- 「估算花費」沿用既有 point-in-time 計費；未定價 model 的標示行為與既有用量頁一致。
- 重疊（成員多 tag）是**刻意接受**的語意，不是 bug——靠 UI 明確標示讓 admin 正確解讀，而非試圖去重。
- 本功能為 admin 治理視角，不下放給成員自助；成員仍只能看自己的整體用量（既有階段 9 能力）。
- 首頁「Top 5 tags by spend」卡片**不在本功能範圍**——它依賴階段 14 的圖表基建，延到階段 14 一起做。
- 既有用量頁已有 CSV/JSON 匯出與時間區間選擇；本功能延伸這些既有能力到 tag 維度，不另造新機制。
