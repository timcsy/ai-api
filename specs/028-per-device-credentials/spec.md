# Feature Specification: 憑證模型重構（每分配多 per-device 憑證）

**Feature Branch**: `028-per-device-credentials`
**Created**: 2026-06-04
**Status**: Draft
**Input**: User description: "把唯一性從 token 移到分配，一筆分配可掛多把獨立的 per-device 具名憑證，各自可單獨撤回、共用分配額度；新增裝置/憑證清單管理（member 自助 + admin 可見可撤）。切 model/換電腦不連坐，既有 token 零回歸。"

## 背景與問題

目前一筆分配只綁**一把** token，重新產生（rotate）會把那唯一一把換掉。這導致：

- **多裝置很痛**：成員在多台電腦用同一把 token；一旦 rotate，**所有裝置同時失效**，每台都得重設。
- **撤銷粒度太粗**：某台電腦遺失，只能 rotate 整把（害到其他裝置），無法「只撤那一台」。
- **忘記複製即卡死**：token 一次性顯示，忘了就只能 rotate。

平台原則本就主張「每筆分配發行**獨立、可識別、可撤回**的憑證」「**撤銷單一憑證不影響其他憑證**」——但現行
一分配一憑證下，最後這句形同空話。本功能把**唯一性的單位從 token 移到「分配」**：一筆分配可掛**多把獨立的
per-device 具名憑證**，讓上述原則名副其實，並對齊業界做法（GitHub PAT、AWS IAM 雙鑰、各家「已授權裝置」清單）。

額度、歸戶、可追蹤性仍綁在**分配**層（多把憑證共用同一額度），故 token 數量不會繞過配額與異常偵測。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 成員為多台裝置各發一把憑證（Priority: P1）

成員在自己的儀表板，為一筆分配新增多把**具名**憑證（如「我的筆電」「公司桌機」），每把都能獨立呼叫本平台，
全部計入同一分配的額度。

**Why this priority**: 這是整個重構的核心價值——多裝置各自一把、不再共用一把；沒有它，後面的獨立撤回也無從談起。

**Independent Test**: 對同一分配新增兩把具名憑證 → 兩把都能成功呼叫 → 兩次呼叫都計入該分配的用量。

**Acceptance Scenarios**:

1. **Given** 一筆 active 分配，**When** 成員新增一把具名憑證，**Then** 顯示一次新 token（可複製），之後該 token 可正常呼叫本平台。
2. **Given** 同一分配已有多把憑證，**When** 用任一把呼叫，**Then** 成功，且用量／費用**歸到該分配**（與用哪一把無關）。
3. **Given** 新增憑證，**When** 檢視清單，**Then** 看到每把的裝置名、建立時間、最後使用時間（若有）。

---

### User Story 2 - 逐把獨立撤回，不影響其他（Priority: P1）

成員（或管理員）可撤回某一把憑證；被撤的那把立即失效，**同分配的其他憑證照常可用**。

**Why this priority**: 「撤銷單一憑證不影響其他」是原則 1 的核心承諾，也是多裝置安全的關鍵（掉一台只撤那台）。與 US1 同為 P1。

**Independent Test**: 一分配有兩把憑證 → 撤回其中一把 → 那把呼叫被拒、另一把仍成功。

**Acceptance Scenarios**:

1. **Given** 一分配有 A、B 兩把憑證，**When** 撤回 A，**Then** A 立即失效（後續呼叫被拒）、B 仍正常。
2. **Given** 成員操作，**When** 嘗試撤回**他人**的憑證，**Then** 被拒（嚴格擁有者檢查）。

---

### US3 - 管理員可見並管理成員的所有憑證（Priority: P2）

管理員可檢視某成員（或某分配）名下的所有憑證，並逐把撤回。

**Why this priority**: 治理需要單一真理路徑——admin 必須看得到、撤得掉。建立在 US1/US2 之後。

**Independent Test**: 管理員開某分配 → 看到其名下所有憑證 → 撤回其中一把 → 該把失效、其他不受影響。

**Acceptance Scenarios**:

1. **Given** 某成員多把憑證，**When** 管理員檢視，**Then** 列出全部（裝置名、前綴、建立/最後使用、狀態）。
2. **Given** 管理員撤回某把，**When** 完成，**Then** 該把失效並留稽核紀錄；其他不受影響。

---

### Edge Cases

- **既有 token（重構前）**：升級後**必須照常可用**（視為該分配的一把預設具名憑證），呼叫零中斷。
- **同一 model 多個分配**：成員可有「Codex 用」「腳本用」等多個分配，各自獨立額度與憑證集合。
- **撤回最後一把**：分配可暫時沒有可用憑證（成員自行再新增）；不影響分配本身狀態。
- **撤回 vs 撤銷分配**：撤回單把憑證 ≠ 撤銷整筆分配（後者沿用既有行為，含自助領取的 reclaim 鎖）。
- **大量裝置**：不設硬性上限（額度在分配層已擋濫用）；清單須能清楚呈現多把。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 允許一筆分配同時擁有**多把有效憑證**，每把有獨立識別與**裝置名（label）**。
- **FR-002**: 每把憑證 MUST 可**單獨撤回**；撤回一把**不得影響**同分配的其他憑證。
- **FR-003**: 不論用該分配的哪一把憑證呼叫，用量／費用／可追蹤性 MUST **歸屬到該分配**（與既有行為一致）。
- **FR-004**: 既有（重構前）的 token MUST 在升級後**照常可用**，呼叫**零中斷**（視為該分配的一把預設具名憑證）。
- **FR-005**: 成員 MUST 能在自己的介面**自助新增／撤回自己的**憑證；**不得**操作他人的憑證（嚴格擁有者檢查）。
- **FR-006**: 新增憑證時，token 明文 MUST **僅顯示一次**且平台**只存雜湊（不可還原）**，沿用既有安全模型。
- **FR-007**: 管理員 MUST 能檢視某成員／分配名下**所有**憑證並逐把撤回；撤回 MUST 留稽核紀錄。
- **FR-008**: 介面 MUST 呈現每把憑證的：裝置名、token 前綴、建立時間、最後使用時間（若有）、狀態。
- **FR-009**: token 數量 MUST **不繞過**分配的配額與異常偵測（多把共用同一分配額度）；**不設**人為數量上限。
- **FR-010**: 本功能 MUST 不改變既有「分配」層的語意（額度、狀態、撤銷分配、自助領取 reclaim 鎖等照舊）。

### Key Entities

- **分配（Allocation）**：唯一性／身分／額度／歸戶的單位（成員 + model + 用途）。一筆分配可有**多把**憑證。既有欄位語意不變。
- **憑證（Credential）**：一把可呼叫的 token 的記錄。**改為一分配多筆**：新增「獨立識別」「裝置名（label）」「最後使用時間」；
  保留 token 雜湊（驗證用）、前綴。每筆可獨立撤回。**呼叫紀錄仍以分配歸戶**（憑證不持有額度）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一筆分配可註冊 **≥2 把**具名憑證，每把都能成功呼叫；用量皆計入該分配。
- **SC-002**: 撤回某一把後，該把呼叫被拒、**同分配其他憑證 100% 仍可用**（0 起連坐失效）。
- **SC-003**: 既有 token 升級後**成功率 100%**、呼叫**零中斷**（migration 不使任何既有 token 失效）。
- **SC-004**: 成員無法取得或撤回他人憑證——**0 起**跨成員操作（以測試固化）。
- **SC-005**: 管理員可列出並撤回某成員所有憑證；撤回皆有稽核紀錄。
- **SC-006**: 既有 proxy／計費／配額／自助領取行為**零回歸**（全套測試綠、含 Postgres 整合）。

## Assumptions

- **唯一性在分配層（硬約束）**：對齊已更新的原則 1——憑證可多把、各自獨立可撤、額度/歸戶在分配層。
- **維持 hash-only**：憑證仍 show-once + 只存雜湊（不做可還原、不做配對碼——屬其他取捨）。
- **device-flow 不在本階段**：RFC 8628 瀏覽器授權拉 token 放階段 19（與 Codex 安裝腳本一起）；本階段新增裝置以
  「顯示一次 + 複製」即可。
- **複用既有機制**：token 產生/雜湊/驗證（`lookup_by_token`）、自助領取/admin 建立分配、稽核紀錄、撤回流程皆複用；
  把 admin 能力下放成員時沿用「同一 service + 端點層擁有者檢查」。
- **既有資料遷移**：升級時把每筆既有憑證原樣保留為該分配的一把預設具名憑證（如名為「預設」），零既有 token 失效。
- **無硬性數量上限**：不設「每分配 ≤N 把」（額度層已擋濫用）。
