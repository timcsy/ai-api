# Quickstart: admin 依模型種類測試模型（手動驗收）

前置：起後端 + 前端、登入 admin、目錄各備一個 對話 / embedding / TTS / 圖片 / STT 模型，且各自供應商有可用憑證。

## US1 — 對話模型（P1 🎯）
1. 進一個對話模型詳情頁，按「測試模型」。
   - ✅ 顯示「測試通過（延遲 N ms）」；無額外確認步驟。
2. 把該供應商憑證改壞（或用不存在的 deployment）再測。
   - ✅ 顯示帶上游原因的失敗；HTTP 非 5xx、頁面不崩。
3. 一個供應商沒有憑證的模型按測試。
   - ✅ 顯示「沒有可用憑證」說明。

## US2 — embedding 模型（P2）
1. 進一個 embedding 模型（litellm mode=embedding）詳情頁，按「測試模型」。
   - ✅ 直接打、顯示通/不通；不被誤判成對話。
2. （已知限制）手動建立、無 litellm 對接的 embedding 模型按測試。
   - ✅ 會以「對話」測法打 → 自然失敗給可辨識訊息（非靜默誤判）。

## US3 — 計費種類 + 費用確認（P2）
1. 進一個圖片生成模型，按「測試模型」。
   - ✅ 先跳「此測試會產生一次實際費用」確認框。
2. 按取消。
   - ✅ 不打任何呼叫、不產生費用（後端也未被呼叫）。
3. 按確認。
   - ✅ 打最小尺寸生圖、顯示通/不通。
4. TTS 模型同 1–3（短文字→語音）。
5. （後端防呆）直接對 billable 模型 `POST .../test` 不帶 `acknowledge_billable`。
   - ✅ 回 `needs_confirmation:true`、**未**打上游。

## US4 — 未支援種類（P3）
1. 進一個 STT（語音轉文字）模型，按「測試模型」。
   - ✅ 顯示「此類型尚不支援自動測試」；不打呼叫、不崩、非無資訊錯誤。

## 不重複 / 零回歸（FR-008 / SC-005）
1. 既有「測試 responses」按鈕仍在、行為不變（兩顆按鈕並列各答各的）。
2. 測試後查 audit → 有 `model_tested` 紀錄（kind/ok/latency，歸戶 admin）；成員/平台用量視圖**不**因測試多出成員用量列。
3. `alembic heads` 不變、無新 migration；`pip` 依賴無新增。
