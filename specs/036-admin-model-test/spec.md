# Feature Specification: admin 依模型種類一鍵測試模型是否可用

**Feature Branch**: `036-admin-model-test`
**Created**: 2026-06-08
**Status**: Draft
**Input**: 在管理員「模型詳情頁」讓 admin 對該模型按一顆「測試模型」按鈕，平台**依模型種類**打一個對應的最小真實呼叫，結果即答案（沿用既有「測試連線 / 測試 responses」的「結果即回應、NEVER 5xx」模式）。v1 涵蓋四種：對話 chat、embedding、TTS、圖片生成；會計費的種類（圖片、TTS）按鈕前明示會產生實際費用並需確認。responses/Agent 相容測試（階段 25）不重複；STT 此版不做。

## 背景與問題

admin 把一個模型加進目錄後，**無法在同一頁確認「這個模型現在到底打不打得通」**。目前只有：(1) 階段 25 的「測試 responses」——只測 `/v1/responses` 那條路；(2) 供應商憑證頁的「測試連線」——測的是「憑證＋模型」組合、入口不在模型頁且只打 chat。對 embedding / 圖片生成 / TTS 這類**非對話模型**，admin 完全沒有就地驗證的方法，只能等成員回報壞掉。

**核心想法**：模型有不同種類，**測試方式也該不同**——對話模型打對話、向量模型打 embedding、語音模型合成一小段、圖片模型生一張最小圖。一顆按鈕，平台依模型種類自動挑對的測法，結果即答案。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 對話模型一鍵測試（Priority: P1）🎯 MVP

admin 在對話類模型（文字輸入、文字輸出）的詳情頁按「測試模型」，平台打一個極小的對話呼叫，通就顯示通過 + 延遲，不通顯示帶上游原因的錯誤。

**Why this priority**: 對話是最大宗、最常用、且幾乎免費可測——立即讓 admin 在同一頁確認「這個模型現在能用嗎」，不必繞去供應商頁。

**Independent Test**: 對一個可用的對話模型按「測試模型」→ 顯示通過 + 延遲；對一個壞掉的（憑證失效 / 部署名錯）→ 顯示帶上游原因的失敗，且不是無資訊錯誤、頁面不崩。

**Acceptance Scenarios**:

1. **Given** 一個對話模型（modality 文字→文字）且其供應商有可用憑證，**When** admin 按「測試模型」，**Then** 平台打一個極小對話呼叫並顯示「通過 + 延遲」。
2. **Given** 同上但上游會失敗，**When** 按「測試模型」，**Then** 顯示帶上游原因的失敗訊息（結果即回應，HTTP 不回 5xx）。
3. **Given** 該模型的供應商沒有可用憑證，**When** 按「測試模型」，**Then** 顯示「沒有可用憑證」的清楚說明，而非崩潰或無資訊錯誤。

---

### User Story 2 - embedding 模型測試（Priority: P2）

admin 在向量（embedding）模型詳情頁按「測試模型」，平台用一個短字串打 embedding，結果即答案。

**Why this priority**: embedding 也幾乎免費可測，是僅次於對話的常見種類；補齊後「免費可測的兩種」都覆蓋。

**Independent Test**: 對一個 embedding 模型按「測試模型」→ 通顯示通過 + 延遲、不通顯示上游原因。

**Acceptance Scenarios**:

1. **Given** 一個 embedding 模型（輸出為向量），**When** admin 按「測試模型」，**Then** 平台以短字串打 embedding 並顯示通/不通。

---

### User Story 3 - 會計費種類（TTS、圖片生成）測試 + 費用確認（Priority: P2）

admin 在 TTS（輸出語音）或圖片生成（輸出圖片）模型詳情頁，按「測試模型」前平台**明示這會產生一次實際費用**並要 admin 確認，確認後才打對應的最小呼叫（TTS：短文字→語音；圖片：最小尺寸生一張）。

**Why this priority**: 這兩類無法「免費試打」——測一次就是真花錢，所以必須先讓 admin 知情並同意，避免誤觸成本。

**Independent Test**: 對 TTS / 圖片模型按「測試模型」→ 先出現「會產生實際費用」的確認；取消則不打、不計費；確認則打最小呼叫並顯示通/不通。

**Acceptance Scenarios**:

1. **Given** 一個圖片生成或 TTS 模型，**When** admin 按「測試模型」，**Then** 先顯示「此測試會產生一次實際費用」的確認提示。
2. **Given** 上述確認提示，**When** admin 取消，**Then** 不打任何呼叫、不產生費用。
3. **Given** 上述確認提示，**When** admin 確認，**Then** 平台打對應種類的最小呼叫並顯示通/不通（圖片＝最小尺寸、TTS＝極短文字）。

---

### User Story 4 - 未支援自動測試的種類給清楚說明（Priority: P3）

對此版尚未支援自動測試的模型種類（如語音轉文字 STT 需範例音檔），「測試模型」顯示清楚說明而非壞掉或誤導。

**Why this priority**: 完整覆蓋「按鈕在所有模型上的行為」，避免某些種類按了沒反應或報怪錯。

**Independent Test**: 對一個 STT（語音輸入）模型按「測試模型」→ 顯示「此類型尚不支援自動測試」之類的清楚說明，頁面不崩。

**Acceptance Scenarios**:

1. **Given** 一個此版未支援的種類（如 STT），**When** admin 按「測試模型」，**Then** 顯示清楚的「尚不支援自動測試」說明，不打呼叫、不報無資訊錯誤。

---

### Edge Cases

- **無法判定種類**：若模型的種類資訊不足以判定，視為對話（最常見、最安全的免費測法）或給「尚不支援」說明，二擇一且行為一致、不崩。
- **多模態模型**（同時吃圖文）：以「輸出種類」為主判定測法（輸出文字→對話、輸出圖片→圖片生成…）。
- **上游真的不支援該呼叫**：回帶上游原因的錯誤（非靜默、非無資訊 5xx）。
- **測試成本歸屬**：測試呼叫是真實呼叫；其費用/用量如何記錄要明確（見 FR-007），不可變成「沒人付的影子用量」。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 模型詳情頁 MUST 提供一顆「測試模型」動作；按下後平台 MUST **依模型種類**選擇對應的最小真實測試呼叫。
- **FR-002**: 模型種類 MUST 由既有的模型中繼資料（輸入/輸出型態）判定，對**所有**模型（含手動建立、無外部登錄表來源者）皆可判定；不得要求新欄位。
- **FR-003**: v1 MUST 支援四種測法：對話（文字→文字，極小呼叫）、embedding（短字串）、TTS（極短文字→語音）、圖片生成（最小尺寸一張）。
- **FR-004**: 對**會產生實際費用**的種類（圖片生成、TTS），平台 MUST 在實際呼叫前明示「會產生一次實際費用」並取得 admin 確認；取消則不呼叫、不計費。免費種類（對話、embedding）MUST 直接呼叫、不額外確認。
- **FR-005**: 測試 MUST 沿用既有「結果即回應」模式：通顯示通過 + 延遲、不通顯示帶上游原因的錯誤；對上游錯誤 MUST NOT 回 5xx（測試結果本身就是回應）。供應商無可用憑證 MUST 給清楚說明。
- **FR-006**: 對此版未支援的種類（如 STT），「測試模型」MUST 顯示清楚的「尚不支援自動測試」說明，MUST NOT 崩潰或回無資訊錯誤。
- **FR-007**: 測試呼叫的費用/用量歸屬 MUST 明確且一致（記為可辨識的 admin 測試呼叫，或沿用既有計費路徑並可被辨識）；MUST NOT 產生無歸屬的影子用量。
- **FR-008**: responses/Agent 相容測試（階段 25 既有「測試 responses」）MUST 不被本功能重複或取代；本功能只補其餘種類。
- **FR-009**: 計費、proxy 其餘行為、既有模型/成員端 MUST 零回歸；**無新表、無 migration、無新套件**。

### Key Entities *(include if feature involves data)*

- **模型目錄項（既有 `ModelCatalog`）**：以其既有「輸入/輸出型態」與（若有）外部登錄表來源的型別資訊判定**種類**；本功能只讀不改其 schema。
- **模型種類（概念）**：對話 / embedding / TTS / 圖片生成 / （未支援：STT…）——由輸出型態為主判定，決定測法與是否需費用確認。
- **測試結果（暫態）**：通/不通 + 延遲 + （失敗時）上游原因；不落新表，即時回應。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: admin 在模型詳情頁，**不離開該頁**即可對對話、embedding、TTS、圖片生成四種模型各自得到「通/不通 + 原因」的結果。
- **SC-002**: 對會計費的種類（圖片、TTS），admin 在實際花費前**必定**看到費用提示並能取消；取消後**零**實際呼叫。
- **SC-003**: 測試免費種類（對話、embedding）時，從按下到看到結果**無**額外確認步驟。
- **SC-004**: 未支援種類（STT）按「測試模型」時，100% 顯示清楚說明、**零**崩潰或無資訊錯誤。
- **SC-005**: 計費與上線前一致；proxy／模型／成員端零回歸；無新 migration、無新套件；測試用量無「無歸屬影子用量」。

## Assumptions

- **沿用既有測試模式**：建立在既有「測試連線（1-token ping、結果即回應、不 5xx）」與階段 25「測試 responses」之上；依模型 slug 解出對應供應商憑證後呼叫。
- **種類判定靠既有輸入/輸出型態**（所有模型都有；外部登錄表的型別資訊當輔助），不新增欄位。
- **多 provider 橋接既有**：對話/embedding/語音/圖片各呼叫由既有多 provider 抽象層提供（library form），不新增套件。
- **最小呼叫**：對話 1-token、embedding 短字串、TTS 極短文字、圖片最小尺寸；皆由 admin 明確觸發，不在成員熱路徑自動測。
- **範圍邊界**：responses/Agent（階段 25 已做）、STT（需內建範例音檔）不在 v1；rerank/moderation 等更冷門種類亦不在 v1。
- **平台/技術棧沿用既有，不新增套件、不新增表、不新增 migration。**
