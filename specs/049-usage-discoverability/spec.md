# Feature Specification: 「如何呼叫」可發現性重設計

**Feature Branch**: `049-usage-discoverability`
**Created**: 2026-06-27
**Status**: Draft
**Input**: User description: "如何呼叫可發現性重設計：金鑰頁如何使用 + 應用商店總站 + cross-link（單一 spec，方案 C，下拉＝這把金鑰 scope 內的 model）"

## 背景與問題

成員（學生）回報「**如何呼叫**」找不到。現況的 API 使用範例**只出現在「分配詳情」「模型目錄詳情」兩個詳情頁**——成員必須先鑽進某個分配或某個模型才看得到；頂層的「金鑰／分配」清單頁沒有任何「怎麼用／開始呼叫」的入口或字眼。

結果是**原則 6 可達性的反面**：平台把能力（金鑰＋可用模型）發給成員了，但成員**找不到怎麼用**——名義上可達、實質上難用，尤其對不熟 API 的非技術成員。本功能把「怎麼用」放到成員**剛拿到金鑰的當下**就看得見的地方。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 在「金鑰」頁就看到「如何使用這把金鑰」（Priority: P1）🎯 MVP

成員拿到一把應用金鑰後，直覺會問「**這把鑰匙怎麼用？**」。在「金鑰」頁，每把金鑰旁就有一個顯眼的「**如何使用這把金鑰**」區：成員從一個**下拉選一個模型**（選項＝這把金鑰實際可用的模型），就立刻看到一段**可直接複製**的呼叫範例（含平台端點、金鑰佔位、以及該模型的正確識別字），不必鑽進任何詳情頁。

**Why this priority**: 這是最高槓桿、最貼「拿到金鑰→怎麼用」當下的一步，直接解掉學生的痛。單獨交付即為可用 MVP。

**Independent Test**: 用一個有 1+ 可用模型的金鑰，開「金鑰」頁 → 看到「如何使用這把金鑰」→ 選一個模型 → 得到含端點、金鑰佔位、正確模型識別字的可複製範例；不需進任何詳情頁。

**Acceptance Scenarios**:

1. **Given** 成員有一把可用 `gpt-5.4-mini` 的金鑰，**When** 在金鑰頁的「如何使用」選 `gpt-5.4-mini`，**Then** 立刻顯示一段可複製的呼叫範例，端點與模型識別字正確、token 以佔位符呈現。
2. **Given** 一把金鑰 scope 內有多個模型，**When** 開啟下拉，**Then** 列出**這把金鑰可用的全部模型**（各有可讀的顯示名稱），切換選擇即更新範例。
3. **Given** 切換到**另一把**金鑰，**When** 看其「如何使用」，**Then** 下拉只列**那把金鑰**可用的模型（清單隨金鑰改變）。
4. **Given** 選到的是非對話類模型（如 OCR／向量／即時字幕），**When** 看範例，**Then** 範例對應**該模型類型的正確端點與形態**，不是對話範例。

---

### User Story 2 - 在「應用」頁找到「怎麼用這個平台」的總站（Priority: P2）

成員想知道「我能拿這個平台接哪些工具、或自己寫程式怎麼接」。「應用」頁成為**「怎麼用」的單一總站**：列出**可接的工具**（如 Codex，各一張卡含設定步驟）以及一張**「直接用 API／SDK」**卡（給要寫程式的人）。

**Why this priority**: 把「用法」收斂到一個可從頂層導覽到達的地方；依賴 P1 的範例內容（單一來源、重用）。

**Independent Test**: 開「應用」頁 → 同時看到「工具整合」卡（至少 Codex）與「直接用 API／SDK」卡，各自有可操作的設定／範例。

**Acceptance Scenarios**:

1. **Given** 成員開「應用」頁，**When** 瀏覽，**Then** 看到「工具整合」與「直接用 API／SDK」兩類用法，工具卡能展開設定步驟、API 卡能取得可複製範例。

---

### User Story 3 - 各處都有通往「怎麼用」的入口（Priority: P3）

成員不論在哪個頁面想起「我要怎麼用」，都找得到去處。**儀表板**（剛拿到金鑰的待辦）、**分配詳情**、**模型詳情**都提供明顯的 cross-link 指向「如何使用／應用」。

**Why this priority**: 補滿資訊氣味（scent），讓任何入口都連得到用法；依賴 P1／P2 的目的地。

**Independent Test**: 在儀表板、分配詳情、模型詳情各自找到一個通往「如何使用／應用」的明顯連結。

**Acceptance Scenarios**:

1. **Given** 成員在儀表板且已有金鑰，**When** 看待辦／快速接入區，**Then** 有一個「開始呼叫／怎麼用」的入口連到「如何使用／應用」。
2. **Given** 成員在分配詳情或模型詳情，**When** 看頁面，**Then** 既有的模型專屬範例保留，並多一句「想接工具 → 看應用」之類的連結。

---

### Edge Cases

- **金鑰沒有任何可用模型**（scope 空、或可用模型全被撤回）→「如何使用」區給**可理解的提示**（去領取／已被撤回），**不顯示壞掉的範例**。
- **一把金鑰含多個模型** → 下拉列全部該金鑰可用模型（不是只第一個、也不是成員的全部模型）。
- **非對話類模型** → 範例對應正確端點（如向量／OCR／即時字幕各自形態）。
- **隔離**：成員只看得到**自己的**金鑰與其可用模型（沿用既有資料隔離）。
- **金鑰只顯示一次**：範例一律用**佔位符**代替真實 token，不重新顯示金鑰明文。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 「金鑰」頁 MUST 對每把金鑰直接顯示「如何使用這把金鑰」區——成員**不需鑽進任何詳情頁**即可看到。
- **FR-002**: 「如何使用」區 MUST 提供**模型選擇**，選項＝**這把金鑰實際可用的模型**（該金鑰被授予的分配對應的模型），而非成員的全部模型。
- **FR-003**: 選定模型後 MUST 顯示**可直接複製**的呼叫範例，內容含：平台端點、金鑰佔位符、以及**該模型的正確識別字**（可被客戶端／程式直接採用）；範例**對應該模型類型的正確端點與形態**（對話／responses／向量／OCR／即時字幕…）。
- **FR-004**: 「應用」頁 MUST 成為「怎麼用這個平台」的總站，至少含**「工具整合」**（≥1 個工具，現為 Codex）與**「直接用 API／SDK」**兩類用法。
- **FR-005**: 儀表板（剛拿到金鑰的待辦）、分配詳情、模型詳情 MUST 各自提供**通往「如何使用／應用」的明顯入口**。
- **FR-006**: 呼叫範例內容 MUST **單一來源、各處重用**（不在多頁各自維護一份），確保措辭／端點／佔位符一致、不會 drift。
- **FR-007**: 入口與標題 MUST 用白話、**喊得出「怎麼用／開始呼叫」**，不只靠「應用」一詞撐起資訊氣味。
- **FR-008**: 一把金鑰 scope 含多個模型時，下拉 MUST 列出**全部該金鑰可用的模型**，各有可讀顯示名稱。
- **FR-009**: 金鑰目前無任何可用模型時，「如何使用」區 MUST 顯示可理解的提示，**不顯示壞掉的範例**。

### Key Entities *(include if feature involves data)*

- **應用金鑰（Application Key）**：成員拿來連線的鑰匙；其 scope ＝一組分配（＝一組可用模型）。「如何使用」以**單把金鑰**為對象。
- **分配（Allocation）**：決定金鑰可用哪些模型，並提供模型的顯示名稱。
- **呼叫範例（Usage Example）**：依模型類型（端點）呈現的可複製範例；**單一來源**、各處重用。
- **應用（Application）**：可接的客戶端工具（Codex…）與「直接 API／SDK」，以**註冊表**呈現（加一個用法＝加一筆資料）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 成員在「金鑰」頁（**不鑽詳情**）即可找到「如何使用這把金鑰」並取得可複製範例。
- **SC-002**: 在「如何使用」選一個模型後，範例的端點與模型識別字**正確對應該模型類型**（對話／向量／OCR／即時字幕…各自正確）。
- **SC-003**: 下拉**只列「這把金鑰可用」的模型**；切換到另一把金鑰，清單隨之改變（不混入其他金鑰的模型）。
- **SC-004**: 「應用」頁同時呈現「工具整合」與「直接用 API／SDK」兩種用法，皆可操作。
- **SC-005**: 儀表板、分配詳情、模型詳情**都有**通往「如何使用／應用」的入口。
- **SC-006**: 呼叫範例只有**一處來源**——改一處，所有出現點同步（無重複維護）。
- **SC-007**（質性）：新成員拿到金鑰後，**不需詢問他人**即可完成第一次呼叫。

## Assumptions

- **單一來源範例**：沿用既有的共用呼叫範例呈現（一處內容、各頁重用），不為各頁新寫一份（呼應「同一概念兩份必 drift」）。
- **金鑰只顯示一次**：範例以**佔位符**代替真實 token，不重新顯示金鑰明文。
- **應用以註冊表呈現**：加一個工具／用法＝加一筆資料，不動架構。
- **Copilot 等非主驗客戶端暫不正式上卡**：本功能**不含 Copilot 專屬卡**（待確認其在平台上能日常用——它會額外請求向量／模型清單等——再另議），避免「列了卻處處紅字」。
- **後端資料應已足夠**：金鑰 scope 的可用模型清單、各模型顯示名稱、依模型類型決定端點範例皆為既有能力；若「這把金鑰可用模型」清單需要端點微調，屬實作細節（plan 階段評估）。
- **以前端呈現為主**：預期**無資料模型／migration 變更**；範圍是把既有內容重新擺到對的位置 + 串對的資料來源。
