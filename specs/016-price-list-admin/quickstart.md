# Quickstart：價目表管理 UI 驗收場景

## 前置
- 階段 3a（pricing 後端）+ 階段 5（多 provider catalog）已部署
- admin 已登入；catalog 至少含一個現行模型（如 `azure/gpt-5.4-mini`），且該模型**尚無價目**

## 場景 1：檢視 + 看出未定價（US1 / SC-001）

1. admin 進「觀測 → 價目」
2. 列表以 catalog 模型為主，每列顯示「目前生效」單價或「未定價」
3. 確認 `azure/gpt-5.4-mini`（或現行模型）標示**未定價**
4. 確認舊模型（若有載過 YAML，如 `gpt-4o-mini`）顯示其 5/1 生效單價

**通過判準**：所有未定價的 catalog 模型 100% 被標示，不必查 DB / YAML。

## 場景 2：新增價格、成本不再為 0（US2 / SC-002 / SC-003）

1. 對未定價模型按「新增價格」→ 填 input/output 每 1K 單價、生效日（今天）、來源備註 → 儲存
2. 列表該模型改顯示新價（不再「未定價」）
3. 用該模型發一次 `/v1` 呼叫（在分配/配額內）→ 到「觀測 → 用量」看該筆成本 **> 0**
4. 重複同 (provider, model, 生效日) 再新增 → 被拒 `duplicate_version`
5. 輸入負數單價 → 被拒 `invalid_price`

**通過判準**：2 分鐘內補價並生效；補價後成本 > 0；重複/負數被擋。

## 場景 3：調價不動歷史帳（US2 / SC-004）

1. 某模型已有 5/1 生效價；先用它跑一筆呼叫（記其成本 C1）
2. 新增 6/1 生效的「加倍」價
3. 6/1 後再跑一筆 → 成本約為 C1 的 2 倍（用新價）
4. 重新查第 1 筆（5 月）的歷史成本 → **仍是 C1**（point-in-time 不變）

**通過判準**：歷史呼叫成本零變動；新呼叫用新價。

## 場景 4：歷史檢視 + 相容性回歸（US3 / SC-006）

1. 展開某多版本模型 → 依生效日列出所有版本，標出「目前生效」與「排程生效」（未來日）
2. 用 CLI 匯入一份 YAML（含與 UI 不同的模型）→ 兩者寫同一張表、都顯示在 UI

```bash
uv run pytest -q          # 既有 335 + 新測試全綠（含 point-in-time 計費回歸）
cd frontend && npm test -- --run   # 既有 72 + 新測試全綠
```

**通過判準**：歷史可稽核；CLI 與 UI 並存；既有計費 / pricing / usage 測試零回歸。
