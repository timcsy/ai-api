# Feature Specification: 成員自助用量視覺化（成員端圖表）

**Feature Branch**: `026-member-usage-charts`
**Created**: 2026-06-04
**Status**: Draft
**Input**: User description: "成員自助用量視覺化：讓非管理員的成員在 dashboard 看到自己用量的圖表（每日趨勢 bar + 各 model 花費 donut），嚴格只看自己的資料。"

## 背景與問題

成員目前在 dashboard 已能看到自己的用量總覽——但那是**純數字 + 純文字**（總 tokens、估算花費、呼叫次數、各 model 明細）。管理員端已有視覺化圖表（趨勢、占比），成員端卻只有數字，較難一眼看出「最近用量節奏」與「花費集中在哪個 model」。

目標是讓**非管理員的成員**也能在 dashboard 看到自己用量的圖表，自己掌握自己的消耗，不必等管理員報數。

**鐵律（資料隔離）**：成員**只能看到自己的**用量，**絕不可**看到任何跨成員的聚合或他人的資料。所有圖表一律以登入者本人為範圍。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 成員看見自己的每日用量趨勢（Priority: P1）

成員在自己的 dashboard 看到一張「每日用量/花費」長條圖，呈現自己（跨所有自己的憑證加總）最近一段時間每天的用量，一眼看出消耗節奏。

**Why this priority**: 「我最近用多少、有沒有突然爆量」是成員最關心、最有感的問題；單張趨勢圖即可獨立交付價值。

**Independent Test**: 以一般成員登入，dashboard 出現每日趨勢圖，數值等於該成員自己各憑證當日用量之和；換另一個成員登入只看到自己的數據。

**Acceptance Scenarios**:

1. **Given** 成員有跨多張憑證、多天的呼叫紀錄，**When** 開啟 dashboard，**Then** 每日趨勢圖每一天的值 = 該成員**自己所有憑證**當天用量之和（不含他人）。
2. **Given** 成員可切換顯示「花費」或「token」，**When** 切換，**Then** 圖表改以對應指標呈現。
3. **Given** 新成員或區間內無用量，**When** 檢視，**Then** 圖表顯示友善空狀態，不報錯。

---

### User Story 2 - 成員看見自己各 model 的花費占比（Priority: P2）

成員看到一張「各 model 花費占比」環圈圖，了解自己的花費集中在哪些 model。

**Why this priority**: 接續趨勢之後，第二關心的是「錢花在哪」；資料已存在於現有用量明細，視覺化即可。

**Independent Test**: 以成員登入，環圈圖各扇形 = 該成員自己各 model 的花費占比；總和與既有數字明細一致。

**Acceptance Scenarios**:

1. **Given** 成員用了多個 model，**When** 檢視環圈圖，**Then** 各扇形對應自己各 model 的花費，占比正確、僅含自己資料。
2. **Given** 區間內無計費用量，**When** 檢視，**Then** 顯示友善空狀態。

---

### User Story 3 - 成員可切換時段，圖表一起更新（Priority: P3）

成員可選擇時段（本週／本月／本季／自訂），上方圖表一起依該時段重新呈現。

**Why this priority**: 讓成員能比較不同期間；建立在 US1/US2 圖存在之後。

**Independent Test**: 切換時段，趨勢圖與占比圖同時更新為該區間的自己資料；切換時有載入指示、非空白閃爍。

**Acceptance Scenarios**:

1. **Given** dashboard 上有圖表，**When** 切換時段，**Then** 所有成員端圖表一起以新區間重新查詢並呈現。
2. **Given** 切換中資料載入，**When** 等待，**Then** 顯示載入指示而非空白或誤導的「無資料」。

---

### Edge Cases

- **無任何憑證／無用量的成員**：圖表空狀態，不報錯、不溢出。
- **跨成員隔離**：成員嘗試以任何方式取得他人資料時，系統必須拒絕或只回自己的資料（不得外洩）。
- **未計費用量**（model 無價）：趨勢圖 token 仍可顯示；花費占比對未計費部分以 0 計或標示，不誤導。
- **手機（最小 360px）**：圖表不得溢出畫面（沿用既有 RWD 規範）。
- **大量資料**：長區間的每日趨勢仍可讀（以日為桶）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統 MUST 讓已登入的**一般成員**在自己的 dashboard 看到「每日用量/花費」趨勢圖與「各 model 花費占比」環圈圖。
- **FR-002**: 所有成員端圖表的資料 MUST **嚴格限於登入者本人**——範圍由通過驗證的 session 身分決定，**不得**由用戶端傳入的識別碼決定，**不得**包含任何他人或跨成員聚合資料。
- **FR-003**: 每日趨勢圖每一天的值 MUST = 該成員**自己所有憑證**當天用量之和。
- **FR-004**: 各 model 占比 MUST 反映該成員自己各 model 的花費，且與既有用量數字明細一致。
- **FR-005**: 成員 MUST 能在趨勢圖切換「花費」與「token」兩種指標。
- **FR-006**: 成員 MUST 能選擇時段（本週／本月／本季／自訂），所有成員端圖表一起依該時段更新。
- **FR-007**: 無資料／未計費／新成員情況 MUST 顯示友善空狀態，不報錯。
- **FR-008**: 成員端圖表在手機（最小 360px）MUST 不溢出畫面（沿用既有 RWD 規範）。
- **FR-009**: 本功能 MUST 不改變管理員端既有圖表，且**不得**把任何跨成員聚合（provider 占比、全平台熱度、跨成員 Top 榜）下放給成員。

### Key Entities

（不適用——複用既有用量資料以成員為範圍聚合，不新增資料模型、不新增表。）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一般成員登入後，在 dashboard 即可看到「自己」的每日趨勢圖 + 各 model 占比圖，無需管理員協助。
- **SC-002**: 成員端任一圖表呈現的數據 **100%** 僅含登入者本人；**0 起**他人資料外洩（以隔離測試驗證）。
- **SC-003**: 每日趨勢與 model 占比的總和與既有用量數字明細**一致**（同一區間、同一成員）。
- **SC-004**: 成員可在 3 秒內切換時段並看到圖表更新（含載入指示）。
- **SC-005**: 在 360px 手機與桌機，成員端圖表皆不溢出畫面；既有測試零回歸。
- **SC-006**: 管理員端既有圖表與行為零回歸。

## Assumptions

- **取向（已定）**：第一版做**兩張圖**（每日趨勢 bar + 各 model 占比 donut），**就地升級成員 dashboard 既有的用量總覽區塊**（成員已在該處看用量，最小導覽成本；控制在 ≤2–3 張圖避免擠）。
- **隔離為硬約束**：成員只看自己——沿用「把 admin 能力下放成 self-service：同一聚合邏輯 + 嚴格擁有者檢查」的既有做法，範圍一律取自 session 身分。
- **複用既有基建**：沿用既有圖表元件、色盤、時段選擇器與既有成員用量資料來源；**不導入新套件**。
- **純呈現 + 既有資料**：以成員為範圍聚合既有用量，不新增資料模型／migration。
- **明確排除（留待之後評估）**：per-allocation 占比圖、配額燃燒投影、成員端熱度圖、獨立「我的用量」頁（第一版就地升級 dashboard）。
