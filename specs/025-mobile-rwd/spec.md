# Feature Specification: 行動裝置（手機）體驗強化（RWD）

**Feature Branch**: `025-mobile-rwd`
**Created**: 2026-06-03
**Status**: Draft
**Input**: User description: "行動裝置（手機）體驗強化（RWD）：讓後台與成員端在手機（360–414px）上也順手——消除導覽列塞爆、中文字字斷行、長字串撐破卡片，桌機體驗零回歸。"

## 背景與問題

平台的管理後台與成員端介面目前是「桌機優先」設計，在桌機／平板上順暢；但實際用**手機**操作時體感很差。經一輪系統性稽核，最有感的痛點**不是說明文案太長**，而是：

1. **導覽列塞不下**——頂部主導覽 + 使用者身分（完整 email）+ 操作按鈕硬擠成一橫排，手機寬度放不下，被裁切或把按鈕擠出畫面。
2. **中文「字字斷行」**——當橫排元素在窄螢幕被壓到比內容還窄，中文標籤會一個字一行折成直條（「我的儀表板」→「我／的／儀／表／板」）。
3. **長字串撐破版面**——email、模型代號、端點網址等長字串不換行，把卡片撐破造成整頁水平捲動。
4. **寬資料表難用**——多欄表格（用量、分配、成員、供應商）在手機要左右滑很遠才看得到欄位與操作。

目標是讓非技術使用者在手機上也**順手**，而非僅「能開」。**桌機／平板體驗必須零回歸。**

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 手機上導覽順暢可達（Priority: P1）

管理員或成員在手機上打開系統，能清楚看到導覽、輕鬆到達每一個頁面——導覽不再塞爆、不字字斷行、不被使用者 email 擠掉；放不下時收合成一個可一鍵展開的選單。

**Why this priority**: 這是使用者實際回報的最大痛點，也是「修一次、整站受惠」槓桿最高的部分（含放大全站擠壓的版面邊距根因）。導覽不可用 = 手機上整個系統難以使用。

**Independent Test**: 在 360px 寬的手機（或裝置模擬）逐一點開頂部主導覽與管理員子導覽的每個目的地，確認全部可達、無橫排溢出、無中文字字斷行、email 不擠掉其他控制項。

**Acceptance Scenarios**:

1. **Given** 以管理員身分在 360px 手機開啟首頁，**When** 檢視頂部導覽，**Then** 主導覽各項與身分／登出控制項皆完整可見或收合於可展開選單中，無任一中文項被折成單字直條，整條導覽不溢出畫面。
2. **Given** email 很長（如 30+ 字元）的使用者登入，**When** 在手機檢視頂部列，**Then** email 被截斷／隱藏／移位，不把登出等控制項擠出畫面。
3. **Given** 管理員子導覽（多個頁籤）在手機，**When** 項目放不下一行，**Then** 以可橫向捲動或收合選單方式呈現，每一項皆可點達且文字不字字斷行。

---

### User Story 2 - 頁面內容在手機上不溢出、不擠爆（Priority: P2）

使用者在手機上瀏覽任一頁面，內容（資訊區塊、表單、卡片、清單）都正常折行、長字串不撐破版面，整頁**不需要左右捲動**。

**Why this priority**: 導覽可達之後，第二順位是讓「到達的頁面本身」可讀。涵蓋資訊區塊多欄擠壓、工具列放不下、長 email／代號／網址撐破卡片等高頻問題。

**Independent Test**: 在 360px 手機逐頁捲動每個管理員與成員頁面，確認整頁無水平捲動、無中文字字斷行、無長字串溢出卡片邊界。

**Acceptance Scenarios**:

1. **Given** 任一含多欄資訊區塊或表單的頁面在 360px 手機，**When** 檢視，**Then** 多欄內容堆疊為單欄，標籤與值皆可完整閱讀、不字字斷行。
2. **Given** 含長 email／模型代號／端點網址的卡片在手機，**When** 檢視，**Then** 長字串截斷（完整值可透過提示／複製取得）或自然換行，不撐破卡片、不造成整頁水平捲動。
3. **Given** 頁面頂部有標題 + 多個操作按鈕的工具列，**When** 在手機寬度放不下，**Then** 工具列自動換行，不溢出畫面。

---

### User Story 3 - 寬資料表在手機上可讀可操作（Priority: P3）

使用者在手機上檢視多欄資料表（用量、分配、成員、供應商、價目等），能讀到關鍵資訊並完成該列的主要操作，不需橫向滑很遠。

**Why this priority**: 寬表格是手機體驗最棘手、但相對低頻（管理員多在桌機處理）的部分；放在最後，且需先確定呈現取向。

**Independent Test**: 在 360px 手機開啟每個多欄表格頁，確認關鍵欄位可讀、每列主要操作可在一個螢幕寬內觸及，不需長距離橫向捲動。

**Acceptance Scenarios**:

1. **Given** 多欄資料表在手機，**When** 檢視，**Then** 每一列以**卡片式堆疊**呈現（該列全部欄位以標籤＋值成對顯示），關鍵欄位（對象、狀態、金額／用量、操作）皆可讀。
2. **Given** 某列有多個操作（如供應商列的測試／輪替／停用），**When** 在手機檢視，**Then** 操作集中於可一鍵展開的選單或可在一個螢幕寬內觸及，不需橫向滑到畫面外。
3. **Given** 表格在桌機（≥768px），**When** 檢視，**Then** 維持原本完整多欄表格呈現，零回歸。

---

### Edge Cases

- **超長不可斷字串**：60 字元無斷點的 email、含長路徑的端點網址——必須截斷或強制換行，不得撐破容器。
- **同時多欄又長值**：8 欄表格且儲存格含長 email／代號——需同時處理欄數與字長。
- **極窄裝置**：320px（舊款／小手機）為 best-effort，不保證像素級完美但不得整頁崩版。
- **空資料**：清單／表格無資料時，空狀態提示在手機亦須置中可讀、不溢出。
- **橫向（landscape）手機**：以寬度為準，落在平板斷點時沿用較寬版面即可。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 在支援的最小手機寬度（假設 360px）下，系統的**任一頁面皆不得產生整頁水平捲動**。
- **FR-002**: 主導覽在手機上**所有目的地皆須可達**；當導覽放不下一行時，須收合為可一鍵展開的選單，不得溢出或裁切。
- **FR-003**: 登入使用者的身分顯示（email）在手機上**不得把其他導覽／操作控制項擠出畫面**（須截斷、隱藏或移位）。
- **FR-004**: 文字須在合理邊界折行；**中文標籤不得被壓成「一字一行」的垂直直條**。
- **FR-005**: 長動態識別字串（email、模型代號、指紋、端點／gateway 網址）**不得溢出其容器**——須截斷（完整值可經提示或複製取得）或自然換行，絕不撐破卡片版面。
- **FR-006**: 工具列／操作按鈕列在手機寬度放不下時須**自動換行**，不得溢出畫面。
- **FR-007**: 多欄資訊區塊與表單列在手機上須**堆疊為單欄**，使標籤與值皆可閱讀、不字字斷行。
- **FR-008**: 多欄資料表在手機上須以**卡片式堆疊**呈現——每一列轉為一張上下堆疊的卡片，顯示該列**全部**欄位（標籤＋值成對），每列主要操作可在一個螢幕寬內觸及，不需橫向捲動；桌機（≥768px）維持完整多欄表格。為避免「同一表格兩套版面隨時間 drift」，卡片式呈現須以**單一共用機制**套用到各表格（同一份資料來源驅動「桌機表格／手機卡片」兩種版面），不為每個表格各寫一套。
- **FR-009**: 既有已正確響應的元素（圖表、熱度圖、程式碼／指令範例區塊、型錄卡片網格）在手機上須**維持現有正確行為**，本功能不得使其退步。
- **FR-010**: 桌機／平板（≥768px）版面**零回歸**——較寬寬度下的呈現與行為須與本功能實作前一致。
- **FR-011**: 既有已正確的響應式做法須**保留沿用**，不重做（避免引入回歸）。

### Key Entities

（不適用——本功能為純呈現層調整，不涉及資料模型、不新增實體。）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在 360px 寬度下，**0 個路由**產生整頁水平捲動（目前有多個頁面會）。
- **SC-002**: 手機上**100%** 的主導覽 + 管理員子導覽目的地皆可達。
- **SC-003**: 跨所有受稽核頁面，中文文字呈現為「一字一行」直條的情形為 **0 起**。
- **SC-004**: 在 360px 下，長識別字串（email／代號／網址）撐破卡片的情形為 **0 起**。
- **SC-005**: 首次使用的使用者在手機上可到達任一管理頁面並閱讀其主要內容，**不需縮放（pinch-zoom）或整頁水平捲動**。
- **SC-006**: 桌機零回歸——既有自動化前端測試全數通過，且 ≥768px 的版面目視與實作前一致。
- **SC-007**: 寬資料表上，每列的主要操作可在**一個螢幕寬內**觸及（不需長距離橫向捲動才看得到）。

## Assumptions

- **最小支援寬度**：360px（現代最小主流手機常見值）；320px 為 best-effort、不保證像素級完美但不得崩版。
- **範圍涵蓋全站**：管理員頁面**與**成員端頁面（儀表板、型錄等）皆納入——成員更可能用手機，管理員亦會在手機查看。
- **沿用既有設計系統**：不導入額外 UI 函式庫；採「桌機完整、手機精簡」哲學（寧可收合／堆疊，也不硬塞到字字斷行）。
- **寬表格取向（已定）**：手機採**卡片式堆疊**（每列一張卡、顯示全部欄位）、桌機維持完整表格；以單一共用機制套用至各表格，避免兩套版面 drift。
- **不碰已做對的**：圖表、程式碼範例區塊、型錄卡片網格等既有正確響應式區域，預設不更動。
- **純呈現層**：無後端／API／資料模型／migration 變更。
- **驗證方式**：以手機寬度（瀏覽器裝置模擬與／或真機）實測為準，不以「桌機 build 通過」當驗收。
- **既有共用基建可重用**：既有的共用元件與設計斷點機制將被沿用以套用響應式調整。
