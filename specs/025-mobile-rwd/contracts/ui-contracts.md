# UI 契約：行動裝置（手機）體驗強化（RWD）

本功能無 API 契約（純前端）。以下為**共用前端機制的呈現契約**——它們是跨頁重用的介面，
契約穩定才能避免各頁 drift。對應憲章「契約優先」於 UI 層的落實。

---

## 契約 1：響應式表格機制 `.responsive-table`

**介面**：一個 CSS class + 每身體格的 `data-label` 屬性約定（見 data-model.md）。

**呈現契約**：

| 條件 | 必須行為 |
|------|---------|
| 視窗 `≥ 768px`（md+） | 維持原生多欄 `<table>` 呈現，與本功能實作前**完全一致**（零回歸）。 |
| 視窗 `< 768px`（手機） | 表頭隱藏；每一資料列轉為一張有框、有間距的卡片；每格以「`data-label`：值」成對顯示；該列**全部欄位**可見。 |
| 任一視窗 | 不得產生整頁水平捲動（手機卡片自然堆疊、桌機由既有 `overflow-auto` wrapper 處理）。 |
| 每列主要操作 | 手機卡片內可於一個螢幕寬內觸及（多操作收於 `DropdownMenu`）。 |

**可測斷言**（vitest，先 Red）：
- 套用 `.responsive-table` 的表格，其每個 body `<TableCell>` 皆帶非空 `data-label`。
- 桌機表格結構（`<table>`/`role="table"`）與手機卡片所需 `data-label` 同時存在於 DOM（CSS 控制顯示）。

**非此契約涵蓋**（手動驗收）：實際在 360px 是否視覺正確堆疊——quickstart 清單。

## 契約 2：手機導覽收合

**介面**：`app-shell.tsx` 在 `< md` 呈現漢堡鈕 + `Sheet` 抽屜。

**呈現契約**：

| 條件 | 必須行為 |
|------|---------|
| 視窗 `≥ 768px` | 維持現有頂部橫排導覽（主導覽 + email + 登出），與實作前一致（零回歸）。 |
| 視窗 `< 768px` | 隱藏 inline 主導覽與 email；顯示漢堡鈕；點擊開啟抽屜。 |
| 抽屜內容 | 涵蓋**全部**導覽目的地（主導覽 + 管理員子導覽，依身分）＋ email ＋ 登出，無遺漏。 |
| 任一視窗 | email 不得把其他控制項擠出畫面（手機隱藏／截斷）。 |
| 子導覽（橫向） | 項目不字字斷行（`shrink-0 whitespace-nowrap`）；放不下時可橫向捲動。 |

**可測斷言**（vitest，先 Red）：
- 手機寬度下渲染 `app-shell`，存在可存取的漢堡鈕（如 `aria-label`）。
- 觸發漢堡 → 抽屜開啟 → 抽屜內可查得全部導覽目的地的連結（admin 身分時含 8 個子導覽 + 登出）。
- 桌機寬度下不渲染漢堡鈕、維持 inline 導覽（既有測試不退步）。

## 契約 3：版面不溢出 / 不字字斷行（全頁通則）

非單一元件，而是跨頁通則，作為驗收契約：

| 項 | 必須行為 |
|----|---------|
| 整頁水平捲動 | 360px 下任一路由皆無（SC-001）。 |
| 中文字字斷行 | 任一導覽/標籤不得呈一字一行直條（SC-003）。 |
| 長識別字串 | email/slug/指紋/URL 不撐破卡片（截斷或換行）（SC-004）。 |
| 桌機 | ≥768px 版面目視與實作前一致（SC-006）。 |

**驗收**：quickstart.md 的 360px 逐頁清單（純視覺，手動）＋ 既有 vitest 全綠（桌機行為零回歸）。
