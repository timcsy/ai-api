# Feature Specification: 管理員成員管理批次化 + 安全刪除

**Feature Branch**: `039-member-batch-admin`
**Created**: 2026-06-10
**Status**: Draft
**Input**: User description: "管理員成員管理批次化 + 安全刪除"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 安全刪除單一成員（Priority: P1）

管理員想刪掉一位成員，但這位成員手上還有分配（甚至有用量紀錄）。目前系統會擋下並要管理員「先自行撤回並刪除分配」——而介面上沒有可一次完成的途徑，等於要找工程師下 SQL。本故事讓管理員在成員列表或成員詳情頁按一次「刪除」，系統就會**安全地**一併處理掉該成員名下的分配與憑證、保留歷史用量紀錄（轉為無歸屬的孤兒紀錄），然後刪除成員——全程一個動作完成、不需工程師介入。

**Why this priority**: 這是使用者當下實際撞到的痛點（畫面上「刪除失敗：revoke and delete allocations before deleting member」）。沒有它，管理員根本刪不掉任何用過的成員。它本身就是一個可獨立交付的 MVP。

**Independent Test**: 建立一位有 1 筆 active 分配（且該分配有數筆呼叫紀錄）的成員，管理員按「刪除」→ 成員消失、其分配與憑證一併移除、歷史呼叫紀錄仍存在（但不再歸屬於任何分配，且仍保有當時的呼叫者識別）。

**Acceptance Scenarios**:

1. **Given** 一位有 active 分配的成員，**When** 管理員對其按「刪除」並確認，**Then** 該成員被刪除、其所有分配（含 active 與已撤回）被移除、其憑證被移除，且該成員過往的呼叫紀錄被保留（不再歸屬於任何分配）。
2. **Given** 一位完全沒有任何分配的成員，**When** 管理員按「刪除」並確認，**Then** 成員直接被刪除、不需額外步驟。
3. **Given** 管理員在刪除前的確認視窗，**When** 視窗顯示，**Then** 視窗清楚說明「將連帶撤回並移除 N 筆分配與 M 把憑證；該成員正在使用的金鑰會立即失效；過往用量紀錄會保留供稽核」。
4. **Given** 任一次成員刪除完成，**When** 查閱稽核紀錄，**Then** 有一筆對應的成員刪除事件（含操作者、時間、被刪成員）。

### User Story 2 - 批次刪除多位成員（Priority: P2）

管理員面對一整批（數十至數百位）成員（例如一學期結束、一整批自動註冊進來的學生帳號要清掉），想一次選取多位、一鍵批次刪除，而不是逐筆點。每一位都走與 P1 相同的「安全刪除」流程，且就算其中幾位刪除失敗也不影響其他位，最後給一份「成功幾位、失敗幾位（各自原因）」的摘要。

**Why this priority**: 規模化清理是管理員管得動大量成員的關鍵；但它建立在 P1 的「安全刪除」之上，故次於 P1。

**Independent Test**: 建立 5 位成員（部分有分配），管理員在列表多選其中 4 位 → 批次刪除 → 4 位都被安全刪除、回傳每位的成功/失敗結果；未選的第 5 位不受影響。

**Acceptance Scenarios**:

1. **Given** 成員列表，**When** 管理員勾選多位成員，**Then** 出現批次操作列顯示「已選 N 位」與「批次刪除」動作。
2. **Given** 已選取多位成員，**When** 管理員按「批次刪除」並在確認視窗確認（視窗顯示即將刪除的位數與連帶影響），**Then** 每位都走安全刪除，回傳逐位成功/失敗摘要。
3. **Given** 批次中某幾位因故刪除失敗（例如競態下狀態改變），**When** 批次完成，**Then** 失敗位不影響其他位的刪除，且摘要列出哪幾位失敗、各自原因。
4. **Given** 批次刪除完成，**When** 查閱稽核紀錄，**Then** 每一位成功刪除的成員各有一筆刪除事件。

---

### User Story 3 - 批次預建本地帳號成員（Priority: P3）

管理員想一次替一批人（例如不走學校 OIDC、要用帳密登入的外部協作者）開帳號。管理員貼上一份 email 清單（每行一個），系統逐筆建立 `local_password` 成員並各自產生一條邀請連結，最後回一份摘要：哪些成功（附邀請連結）、哪些已存在、哪些格式不合法。

**Why this priority**: 補足「新增」端的規模化；但多數成員是 OIDC 自助註冊、預建需求較低，故優先序最低。

**Independent Test**: 管理員貼上 3 個 email（1 個全新、1 個已存在、1 個格式錯）→ 提交 → 回摘要：1 筆成功（附邀請連結）、1 筆已存在、1 筆格式錯；對應的 1 位新成員出現在列表。

**Acceptance Scenarios**:

1. **Given** 批次新建輸入框，**When** 管理員貼上多行 email 並提交，**Then** 系統逐筆建立 `local_password` 成員、各自附一條邀請連結，並回逐筆結果摘要。
2. **Given** 清單中有 email 已存在於系統，**When** 提交，**Then** 該筆回報「已存在」、不重複建立、不影響其他筆。
3. **Given** 清單中有格式不合法的 email 或重複行，**When** 提交，**Then** 該筆回報「格式不合法／重複」、其餘合法筆照常建立。
4. **Given** 批次新建完成，**When** 查閱稽核紀錄，**Then** 每一位成功建立的成員各有一筆建立事件。

---

### Edge Cases

- **刪到自己**：管理員不可在批次或單筆刪除中刪除自己的帳號（避免把唯一管理員清掉而鎖死後台）；該筆應被擋下並提示。
- **刪除最後一位管理員**：系統不得讓最後一位 active 管理員被刪除（無論單筆或批次），以免無人能進後台。
- **空輸入**：批次刪除未選任何成員、或批次新建貼空白清單時，給明確提示、不送出。
- **超大批次**：一次選取/貼上極多筆（例如數百）時，系統仍須逐筆獨立處理、回完整摘要，不因單筆失敗中斷整批。
- **競態**：批次處理期間某成員的分配狀態被他人改變、或成員已被刪除，該筆回報失敗原因、不影響其他筆。
- **正在使用中的金鑰**：安全刪除會使該成員 active 分配與憑證立即失效；確認視窗須事先告知，避免管理員誤刪正在服務的帳號。
- **重複 email（同一次批次新建輸入內）**：同一份清單內出現兩次相同 email，僅建立一次、其餘回報重複。

## Requirements *(mandatory)*

### Functional Requirements

#### 安全刪除（P1）

- **FR-001**: 系統 MUST 允許管理員刪除一位仍擁有分配（active 或已撤回）的成員，並在同一個動作內完成：撤回並移除該成員的所有分配、移除其所有憑證、然後刪除成員——管理員無須先手動處理分配。
- **FR-002**: 安全刪除 MUST 保留該成員過往的呼叫／用量紀錄供稽核；這些紀錄在分配被移除後不再歸屬於任何分配，但仍保有當時記錄的呼叫者識別（不得無聲消失）。
- **FR-003**: 單一成員的安全刪除 MUST 為原子操作——若過程中任一步失敗，該成員不得停留在「分配已刪一半、成員還在」的半完成狀態。
- **FR-004**: 系統 MUST 在管理員確認刪除前，清楚呈現連帶影響：將移除的分配數與憑證數、「正在使用的金鑰會立即失效」、以及「過往用量紀錄會保留」。
- **FR-005**: 每一次成功的成員刪除 MUST 寫入一筆稽核事件（含操作者、時間、被刪成員識別）。

#### 批次刪除（P2）

- **FR-006**: 系統 MUST 讓管理員在成員列表多選成員，並對選取的成員執行批次刪除。
- **FR-007**: 批次刪除 MUST 對每一位成員獨立套用 FR-001～FR-003 的安全刪除流程；單一成員失敗 MUST NOT 中止或回滾其他成員的刪除。
- **FR-008**: 批次刪除 MUST 回傳逐位結果摘要：哪些成功、哪些失敗及各自原因。
- **FR-009**: 系統 MUST 在批次刪除執行前要求確認，且確認視窗顯示即將刪除的位數與連帶影響。

#### 批次預建本地帳號（P3）

- **FR-010**: 系統 MUST 讓管理員一次提交多個 email（每行一個），逐筆建立登入方式為「本地帳密（local_password）」的成員。
- **FR-011**: 每一筆成功建立的成員 MUST 各自產生一條邀請連結，並在結果中回傳。
- **FR-012**: 批次新建 MUST 逐筆獨立處理並回逐筆結果摘要，區分：成功（附邀請連結）、已存在、格式不合法／重複；單筆失敗 MUST NOT 影響其他筆。
- **FR-013**: 每一位成功建立的成員 MUST 寫入一筆建立稽核事件。

#### 共通與防呆

- **FR-014**: 系統 MUST 阻止管理員刪除自己的帳號（單筆與批次皆然）。
- **FR-015**: 系統 MUST 阻止刪除最後一位 active 管理員（單筆與批次皆然），確保後台不會變成無人可進入。
- **FR-016**: 所有刪除與批次操作 MUST 僅限管理員執行；非管理員或未驗證者一律被拒。
- **FR-017**: 既有的單筆新增成員、單筆刪除（無分配時）、分配與計費流程 MUST 維持原行為不變（零回歸）。

### Key Entities *(include if feature involves data)*

- **成員（Member）**：被管理的對象；具 email、登入方式、是否管理員、狀態。本功能新增「安全刪除」與「批次預建」兩種操作其生命週期的方式。
- **分配（Allocation）**：成員對某資源（模型）的存取額度；安全刪除時被撤回並移除。是目前阻擋刪除成員的關聯。
- **憑證（Credential）**：成員持有、用來呼叫 API 的金鑰；隨成員刪除一併移除。
- **呼叫紀錄（CallRecord）**：歷史用量／稽核紀錄；安全刪除後保留為「無歸屬分配」的孤兒紀錄，仍保有呼叫者識別。
- **稽核事件（Audit Event）**：每筆成員建立／刪除都會留下的可追溯紀錄。
- **批次操作結果（Batch Result）**：一次批次刪除或批次新建的逐筆成功/失敗摘要（非持久化實體，為操作的回傳）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 管理員能在單一動作內刪除一位「有分配且有用量紀錄」的成員，全程不需工程師或資料庫操作介入（不再出現「請先撤回並刪除分配」的擋阻）。
- **SC-002**: 安全刪除後，被刪成員過往的呼叫／用量紀錄 100% 仍可在系統中查得（保留為孤兒紀錄），無任何遺失。
- **SC-003**: 管理員能一次選取並批次刪除多位成員；當其中部分失敗時，成功的成員仍被刪除，且摘要正確列出每位的成功/失敗與原因。
- **SC-004**: 管理員能貼上一份混合（全新／已存在／格式錯）的 email 清單批次預建本地帳號，系統正確分類每一筆並只對全新且合法者建立成員 + 邀請連結。
- **SC-005**: 任何情況下系統都不會讓管理員刪除自己或最後一位管理員，後台永遠至少有一位可登入的管理員。
- **SC-006**: 既有單筆新增/刪除、分配、計費功能零回歸。

## Assumptions

- **沿用既有單筆邏輯**：批次刪除與批次新建分別以既有「單筆安全刪除」與「單筆建立成員」流程逐筆執行，不另立一套規則（批次＝迴圈＋結果聚合）。
- **孤兒保留即正解**：用量紀錄在分配移除後保留為無歸屬紀錄，不另做「明確清除歷史」選項；呼叫者識別已隨紀錄保存，足以維持稽核能力。
- **批次新建限本地帳號**：批次預建僅針對 `local_password` 登入方式；OIDC 成員仍走自助註冊、不在本功能預建範圍。
- **逐筆獨立、不整批回滾**：批次操作以「每筆獨立成敗」為原則，不採「一筆失敗整批回滾」；單一成員的安全刪除內部則為原子。
- **沿用既有批次多選 UI 樣板**：批次選取與摘要呈現沿用系統既有的批次操作介面慣例，維持一致體驗。
- **不在本功能範圍**：成員角色／權限的批次變更、批次匯入 CSV 檔上傳（以貼上文字清單為主）、以及自動標籤的網域規則（已存在）。
