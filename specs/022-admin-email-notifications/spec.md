**狀態**: Draft

## 使用者情境與測試 *(必填)*

### 使用者故事 1 - 管理員設定通知管道並驗證可用（優先級：P1）

平台管理員打開 admin web UI 的「通知」設定頁，輸入寄信服務的連線資訊（host、port、帳號、密碼、寄件者識別），以及要接收告警的 email 清單。儲存後，按下「發測試信」按鈕，確認測試信在數秒內抵達。如果沒收到，頁面立即顯示清楚的錯誤訊息（authentication 失敗、收件人被拒、host 不可達等）。

**為何此優先級**：在 admin 能端到端確認管道可用前，任何自動告警都不能被信任。這是其他一切建立的基礎——沒有它，自動通知是個黑盒。

**獨立可測**：可單獨完整測試——輸入憑證、儲存、按測試、確認收到（或看到錯誤訊息）。交付價值：「我知道我的通知設定是對的」。

**驗收情境**：

1. **Given** 輸入有效憑證，**When** admin 儲存並按「發測試信」，**Then** 測試信在 30 秒內抵達測試收件人，頁面顯示成功
2. **Given** 輸入錯誤憑證，**When** admin 按「發測試信」，**Then** 頁面在 30 秒內顯示可行動的錯誤訊息（例：「authentication failed」），且不寄出任何信
3. **Given** 先前已存有效憑證，**When** admin 重新載入頁面，**Then** 設定持久化（密碼以遮罩 fingerprint 顯示，非明文），測試鈕可隨時再用
4. **Given** 通知已設定，**When** admin 清除設定，**Then** 平台回到「通知停用」狀態，不影響任何其他 admin 或使用者面向功能

---

### 使用者故事 2 - 分配被自動隔離時管理員收到 email（優先級：P1）

平台的異常偵測器在非辦公時間自動隔離一筆分配。30 秒內，收件人清單上的每個 email 都收到一封信，內容包含：哪筆分配、屬於哪位成員、為何被隔離（例：「過去 1 小時 1100 calls，baseline 100/hr，11× 突增」）、發生時間、以及可直接點擊跳到 admin 解除頁的連結。admin 在手機上讀信，判斷這次突增合不合理，直接處理——完全不必開桌機。

**為何此優先級**：這是本功能的招牌使用案例。下班後被隔離 = 真實使用者被擋直到 admin 手動介入。今天 admin 只能在登入 UI 時看到隔離。消除這個空白就是這個功能存在的全部理由。

**獨立可測**：故事 1 完成後，觸發一次自動隔離（測試 fixture 或真實條件），驗證設定的 recipients 在 30 秒內收到內容符合的信。

**驗收情境**：

1. **Given** 通知已設定且至少一個 recipient，**When** 異常偵測器自動隔離一筆分配，**Then** 每個 recipient 在 30 秒內收到含以下內容的信：分配識別碼、成員名稱、隔離原因（含具體數字）、時間戳、解除頁連結
2. **Given** 通知**未**設定，**When** 同一隔離事件發生，**Then** 平台正常記錄事件（既有 audit log 不受影響）、不噴 error、服務繼續運作
3. **Given** 寄信給某一 recipient 失敗，**When** 事件發生，**Then** 其他 recipients 仍收到信，失敗被記下足以診斷的細節（哪個 recipient、mail server 回什麼）

---

### 使用者故事 3 - 其他重要平台事件管理員也收到 email（優先級：P1）

除自動隔離外，admin 還會收到以下類別事件的信：(a) **短時間 upstream provider 連續失敗**（例：AI provider 在短時間內連續拒絕多筆呼叫——真實 outage 或設定錯誤）；(b) **儲存的 provider 憑證失效**（例：上游 service 撤銷或過期該 key）；(c) **每日上限被觸發**（在 Phase 16 daily-cap 功能啟用時生效）。每封信遵循與隔離信相同形狀：發生什麼、影響誰／哪個、具體數字、時間戳、相關 admin 頁面連結。

**為何此優先級**：只對單一事件型別告警的通知系統會錯過大部分維運事件。這三類事件是近期實際造成使用者抱怨的源頭。

**獨立可測**：分別觸發每一類事件（測試 fixture），驗證對應形狀的信抵達每個 recipient。

**驗收情境**：

1. **Given** 通知已設定，**When** upstream AI provider 在 5 分鐘內拒絕 10 筆以上呼叫，**Then**「upstream provider failures」信在跨過門檻後 30 秒內抵達
2. **Given** 通知已設定，**When** 一筆儲存的 provider credential 下次呼叫時 authentication 失敗，**Then**「invalid provider credential」信在 30 秒內抵達，含哪個 provider 與哪張 credential
3. **Given** 通知已設定，且某筆分配有 daily cap，**When** 該分配在某天首次撞到 cap，**Then**「daily cap reached」信在 30 秒內抵達（且同一分配同一天不再寄第二封）

---

### 使用者故事 4 - 去重避免事件爆量時 email 灌爆信箱（優先級：P2）

事件爆發期（例：行為失常的 CLI 每分鐘送出數百筆 upstream 失敗呼叫），admin 在 5 分鐘窗內每個事件型別最多收到一封信——不管底層觸發了多少次。該封信含累計次數與最近一次的細節，admin 仍能掌握事件規模，但不會被重複告警淹沒。

**為何此優先級**：沒有去重的話，真實事件可灌爆 admin 信箱數百封一樣的信，逼 admin 忽略通知、把有用細節埋掉。沒有去重時管道本身能跑（故事 1+2+3），但真實事件中管道會變得不可用。

**獨立可測**：5 分鐘內觸發 50 筆同型別事件，驗證恰好寄出 1 封信（body 含計數 50）。

**驗收情境**：

1. **Given** 通知已設定且 4 分鐘內發生 50 次隔離事件，**When** 5 分鐘窗結束，**Then** 恰好寄出一封信，內含計數（50）與最近一筆事件細節
2. **Given** 同一個 5 分鐘窗內發生兩種不同事件（一次隔離、一次 upstream burst），**When** 兩個窗都結束，**Then** 寄出兩封不同信——去重是按事件型別、非全域
3. **Given** 5 分鐘窗結束時僅有 1 次事件，**When** 寄信，**Then** 內容含完整單一事件細節，不要出現多餘的「1 次發生」文字

---

### 使用者故事 5 - 管理員檢視最近通知歷史（優先級：P3）

在同一個通知設定頁，admin 可以看到平台最近寄出通知的列表：時間戳、事件型別、recipients、每位 recipient 的寄送結果（成功 / 失敗 / 因停用被略過 / 因去重合併）。協助確認「我應該要有收到那封信」或排查「為什麼我沒收到」。歷史唯讀。

**為何此優先級**：對信任度排查（「信真的沒到，還是根本沒寄」）有用，但不阻擋價值交付——admin 透過故事 1 的測試鈕已可驗證寄送是否成功。

**獨立可測**：觸發若干通知（測試寄或真實事件）後，頁面以正確欄位列出。

**驗收情境**：

1. **Given** admin 今天已觸發 3 次通知，**When** 開啟通知頁，**Then** 歷史以反序顯示 3 筆，含時間戳、recipients、結果
2. **Given** 一次通知因去重被合併，**When** admin 看歷史，**Then** 該筆明確標示合併（例：「49 筆事件合併入此封」）
3. **Given** 發生寄送失敗，**When** admin 看歷史，**Then** 失敗原因以白話顯示（例：「收件人被拒——信箱不存在」）

---

### 邊界情況

- 儲存的密碼解密失敗（例：加密金鑰被輪替）？→ 設定頁顯示「憑證需要重新輸入」、不嘗試任何寄送、平台繼續運作
- recipient 清單空？→ 不寄任何信；UI 顯示「無收件人——告警停用」。允許存空清單（這是「暫停告警但保留 SMTP 憑證」的方式）
- recipient 信箱滿或被拒？→ 該 recipient 寄送失敗、其他 recipients 仍收到、失敗以 mail server 回應內容記入歷史
- mail server port 被網路政策擋？→ 測試寄在 30 秒內回網路錯誤，使用者面向訊息為「無法連線到 mail server」
- 4 個事件型別同時在事件爆發中觸發？→ 寄 4 封信（每型別一封），各自帶自己的去重計數
- 平台在去重窗中重啟？→ 進行中的去重狀態消失；重啟後第一筆事件啟動新窗。（可接受：重啟罕見；重啟後多寄一封信遠比漏寄一封信無害）
- admin 在通知寄送中清除設定？→ 進行中的寄送完成；後續事件找不到設定就被略過
- 全部 recipients 都失敗（整批寄送都掛）？→ 每位 recipient 的失敗各自記錄；不影響觸發事件的 audit 紀錄；平台正常運作

## 需求 *(必填)*

### 功能需求

**設定**

- **FR-001**: 系統 MUST 允許 admin 透過 web admin UI 設定寄信服務的連線憑證（host、port、帳號、密碼、寄件者 email、寄件者顯示名稱）
- **FR-002**: 系統 MUST 允許 admin 設定一份「一或多個 email」的 recipient 清單以接收通知
- **FR-003**: 系統 MUST 以「資料庫被攻破單獨不足以還原明文密碼」的方式儲存帳號密碼
- **FR-004**: 系統 MUST 允許 admin 更新或清除設定，無需重啟任何平台服務
- **FR-005**: 通知設定不存在或被標為停用時，平台 MUST 維持正常運作——沒有啟動錯誤、事件發生不噴錯、audit log 不受影響

**測試寄送**

- **FR-006**: 系統 MUST 在通知設定頁提供「發測試信」操作
- **FR-007**: 系統 MUST 在「發測試信」操作中要求 admin 即時輸入一個一次性測試收件人 email，測試信只寄到該位址。已儲存的 recipient 清單在測試寄送中**不**被使用，以避免設定階段誤 spam 正式 recipients
- **FR-008**: 測試寄送操作 MUST 在 30 秒內回給 admin 清楚的結果：成功，或標明失敗階段的可行動錯誤訊息（authentication / 連線 / 寄件者被拒 / 收件人被拒 / 未知）

**事件觸發（v1 集合）**

- **FR-009**: 系統 MUST 在異常偵測器自動隔離一筆分配時，自動寄通知信
- **FR-010**: 系統 MUST 在 upstream AI provider 在短時間內連續失敗時，自動寄通知信（預設門檻：5 分鐘內 10 次失敗；由平台維運者透過 deploy-time 設定調整，v1 不開放 admin UI 編輯）
- **FR-011**: 系統 MUST 在偵測到儲存的 AI provider 憑證失效時，自動寄通知信
- **FR-012**: 系統 MUST 在分配某天首次撞到 daily cap 時自動寄通知信（僅當 daily-cap 功能上線才生效；否則無事件可觸發）
- **FR-013**: v1 的事件型別清單 MUST 固定於平台 code/config；admin UI 在 v1 **不需要**提供逐事件型別的開關

**信件內容**

- **FR-014**: 每封通知信 MUST 包含：事件型別、時間戳（時區對台灣讀者清楚）、影響的資源識別碼（allocation ID / credential ID 等，並儘量帶 friendly 顯示名）、解釋「為什麼」的具體數字（例：「1100 calls vs baseline 100/hr」）、可點擊跳到對應 admin 頁的連結
- **FR-015**: 信件 subject 行 MUST 在手機通知預覽中可掃讀（約 50 字內），不開信即傳達嚴重程度與事件型別
- **FR-016**: 信件 body MUST 在純文字 client 中可讀（連結可用 HTML，但不渲染 HTML 仍應有用）

**送達與去重**

- **FR-017**: 系統 MUST 在觸發事件後 30 秒內寄出每封通知信（從事件被記錄到 mail server 回應為止）
- **FR-018**: 系統 MUST 對「同事件型別」在 5 分鐘窗內去重：每個事件型別每 5 分鐘窗最多一封信
- **FR-019**: 被去重的信 MUST 含合併入該窗的總次數，與最近一筆事件的細節
- **FR-020**: 同 5 分鐘窗內不同事件型別 MUST 各自產生自己的信——去重是「按事件型別」、非全域
- **FR-021**: 寄給某一 recipient 失敗時，系統 MUST 仍獨立嘗試所有其他 recipients

**歷史與可視性**

- **FR-022**: 系統 MUST 在 admin 通知頁保留最近的通知嘗試紀錄（時間戳、事件型別、recipients、逐 recipient 結果）
- **FR-023**: 系統 MUST 在歷史中清楚標示去重合併的紀錄（例：「49 筆事件合併入此封」）
- **FR-024**: 通知歷史保留至少 30 天

**失敗模式**

- **FR-025**: 寄信失敗（任何原因）時，系統 MUST 記錄失敗：哪個事件、哪個 recipient、mail server 的錯誤回應。失敗 MUST NOT 導致觸發事件的 audit 紀錄寫不進去
- **FR-026**: 密碼解密失敗（例：加密金鑰換了）時，系統 MUST 在設定頁顯示明確「憑證需要重新輸入」、拒絕嘗試任何寄送
- **FR-027**: 系統 MUST NOT 因為寄信失敗而 crash、重啟、或中斷服務

### 主要實體

- **Notification Configuration（通知設定）**: 持有此通知管道的連線資訊與 recipient 清單。每個平台部署一份。含狀態旗標（啟用 / 停用 / 憑證無效）。密碼部分以「資料庫被攻破單獨不足以還原」方式儲存
- **Notification Event Type（通知事件型別）**: 可觸發 email 的事件分類目錄（v1：分配自動隔離、upstream 短時間連續失敗、provider 憑證失效、daily cap 撞到）。v1 此清單固定——可由平台維運者 deploy-time 設定，不由 admin UI 編輯
- **Notification Record（通知紀錄）**: 歷史表中的一列，對應每一次通知嘗試（單一事件或去重合併）。含時間戳、事件型別、recipients、逐 recipient 結果、原始事件參照

## 成功標準 *(必填)*

### 可量測結果

- **SC-001**: 新管理員首次打開通知頁，能在 5 分鐘內完成 SMTP 設定並成功發出一封測試信
- **SC-002**: 至少 95% 的真實自動隔離事件能在異常偵測器決策後 30 秒內讓 admin 信箱收到通知
- **SC-003**: 真實事件中產生 50 筆以上同型別事件時，admin 在每事件型別每 5 分鐘窗內最多收到 1 封信（可事後比對事件數與寄信數驗證）
- **SC-004**: 通知設定不存在或無效時，對平台功能零影響——proxy throughput、登入、所有 admin 頁面照常運作；無任何啟動時錯誤
- **SC-005**: 寄信失敗發生時，admin 在至少 80% 案例能從通知歷史單獨一筆紀錄判斷根因，無需查閱 application log
- **SC-006**: 下班時段自動隔離事件的「平均回應時間」（隔離 audit event 時間戳 → admin 按下解除的時間戳）相比未上線通知前的基線減少至少 50%
- **SC-007**: 至少 90% 的下班時段事件能在發生後 2 分鐘內透過 email app 推播抵達 admin 手機，admin 不必在桌機前

## 假設

- admin 已有可用的 SMTP 寄信服務（例：學校的 Google Workspace 透過 Gmail SMTP + App Password、或學校既有 mail server）。平台本身**不**提供或自架寄信服務
- v1 目標使用者群（台灣學校與小型組織）以 email 作為通知管道即可。其他管道（LINE Bot、Web Push 等）為後續以平行 adapter 加入，不會改 v1 架構
- admin 人數通常少（每部署 1–5 人），單一共享通知設定即足夠。逐 admin 偏好設定不在 v1 範圍
- 哪些事件型別會寄信由平台維運者（deploy-time）決定，非個別 admin。v1 保持簡單並避免關鍵告警被誤關
- 平台既有的 audit-event 記錄是「發生了什麼」的單一真理。通知系統觀察 audit events、不另開事件管線
- 平台已有資料庫敏感欄位加密機制（既用於 provider credentials）。SMTP 密碼沿用既有機制——本功能不引入新的加密基礎建設
- 30 秒內的 email 送達延遲對目標使用情境可接受。次秒級推播（需 Web Push 或行動 app 等不同管道）不在 v1 範圍
- 造成去重的事件爆發夠罕見，所以「合併為一封含計數 + 最近一例」資訊量已足夠；admin 需深入分析時走既有 admin 頁面
- daily-cap 事件觸發（FR-012）僅在 daily-cap 功能（Phase 16）上線後生效；否則無事件可觸發，等同 no-op
