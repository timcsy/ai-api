# Feature Specification: 會員導覽重排——凸顯「應用」

**Feature Branch**: `051-member-nav-reorder`
**Created**: 2026-06-28
**Status**: Draft
**Input**: User description: "會員導覽重排凸顯應用：MAIN_NAV 改為 儀表板→應用→模型目錄→分配→用量→金鑰，桌機+手機同步，標籤與路由不變"

## 背景與問題

會員頂部導覽目前的順序是 **我的儀表板 / 金鑰 / 分配 / 用量 / 模型目錄 / 應用**——最實用、面向「我要拿來把 AI 用起來」的**「應用」被排在最後一個**，躲在金鑰、分配等「水管/設定」類項目後面。這與平台希望「拿到能力的人能在不依賴工程師的情況下真的用起來」（可達性）相違，也跟既有設計把「應用」定位為**接工具的總站**自相矛盾。

心智模型上：**應用＝結果**（在 VS Code / Cursor / 直接用 API 把 AI 接起來用）、**金鑰＝水管**（連線用的憑證）、**分配＝你被准用哪些模型**（管理員給、會員多半只是查看）。把「結果」排在「水管」後面，等於把最該被看見的入口藏起來。

本功能就做一件事：**重排會員導覽，讓「應用」提到前面、金鑰降到最後（但不隱藏）**，桌機與手機一致。標籤文字與所有路由皆不變。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 會員一眼找到「應用」入口（Priority: P1）

會員（含不熟技術的成員）登入後，在頂部導覽的**顯眼前段**就看到「應用」，不必掃到最後一項，也不必先穿過金鑰/分配等設定類項目。

**Why this priority**: 這是本功能的全部價值——把最常用、最能讓人「真的用起來」的入口提前，直接服務可達性。沒有它，功能不存在。

**Independent Test**: 以一般會員身分檢視導覽，確認「應用」出現在第二位（緊接儀表板），且整體順序為 儀表板 → 應用 → 模型目錄 → 分配 → 用量 → 金鑰。

**Acceptance Scenarios**:

1. **Given** 一位已登入的一般會員，**When** 檢視頂部主導覽，**Then** 項目順序為 **儀表板 → 應用 → 模型目錄 → 分配 → 用量 → 金鑰**。
2. **Given** 同一位會員，**When** 檢視導覽，**Then** 「應用」位於第二位（儀表板之後、任何設定類項目之前）。
3. **Given** 導覽上的任一項目，**When** 點擊它，**Then** 仍前往原本對應的頁面（沒有任何連結改變或失效）。

---

### User Story 2 - 桌機與手機看到一致的新順序（Priority: P2）

無論用桌機的橫向導覽或手機的收合選單，會員看到的導覽順序一致（都是新順序）。

**Why this priority**: 平台明確支援手機；若只改一邊會造成兩種心智模型。依賴 US1 的順序定義，故為 P2。

**Independent Test**: 在桌機寬度與手機寬度各檢視一次導覽，兩者項目順序相同且皆為新順序。

**Acceptance Scenarios**:

1. **Given** 桌機寬度，**When** 檢視橫向主導覽，**Then** 順序為新順序。
2. **Given** 手機寬度，**When** 展開收合選單，**Then** 順序與桌機相同（新順序）。

---

### Edge Cases

- **金鑰仍須可達**：金鑰雖降到最後，但仍是導覽項目、路由不變——troubleshooting / 多裝置輪替的使用者要找得到（凸顯應用 ≠ 隱藏金鑰）。
- **管理員導覽不受影響**：管理員專屬項目（如「管理員」入口、admin 子導覽）的位置與內容不在本次調整範圍。
- **深連結零回歸**：任何既有書籤 / 分享連結（指向金鑰、分配、用量、目錄、應用各頁）仍正確開啟。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 會員主導覽 MUST 以下列順序呈現：**儀表板 → 應用 → 模型目錄 → 分配 → 用量 → 金鑰**。
- **FR-002**: 「應用」MUST 位於主導覽第二位（緊接儀表板，且在所有設定/憑證類項目之前）。
- **FR-003**: 導覽各項目的**顯示標籤文字 MUST 不變**（含「應用」維持原字樣）。
- **FR-004**: 導覽各項目的**目的地（路由）MUST 不變**；不得新增、移除或改寫任何路由，既有深連結零回歸。
- **FR-005**: 桌機與手機（收合選單）MUST 呈現**相同的**新順序。
- **FR-006**: 「金鑰」MUST 仍為導覽項目（位於最後），不得被移除或隱藏。
- **FR-007**: 管理員專屬導覽項目的相對位置與可見性 MUST 不受本次調整影響。

### Key Entities

*(本功能不涉及資料模型——純呈現層導覽順序調整，無新增/變更任何資料。)*

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 會員主導覽順序為 儀表板 → 應用 → 模型目錄 → 分配 → 用量 → 金鑰；「應用」在第二位。
- **SC-002**: 桌機與手機兩種版型的導覽順序 100% 一致。
- **SC-003**: 所有既有路由 / 深連結零回歸（六個會員頁與管理員入口皆照舊開啟）。
- **SC-004**: 導覽標籤文字與本次調整前完全相同（無任何改字）。

## Assumptions

- **僅重排、不改標籤**：經與維護者確認，「應用」標籤維持原字樣（不改成「開始使用」等），本次只動順序。
- **儀表板仍為第一**：維持儀表板為首項（不做 apps-first 落地頁——那是另議的後續）。
- **純前端呈現層**：導覽順序來自單一前端清單來源，桌機與手機共用之；後端、資料模型、路由皆不動。
- **建金鑰捷徑等不在此範圍**：把建金鑰流程整個搬進「應用脈絡」屬後續評估，不在本功能。
- 對應願景**階段 37**（會員 IA 重排凸顯「應用」第一刀）。
