# Feature Specification: 會員介面分頁化 + 金鑰/分配 概念釐清

**Feature Branch**: `032-member-ui-tabs`
**Created**: 2026-06-05
**Status**: Draft
**Input**: 把會員「一頁長捲」拆成頂部導覽分頁（儀表板/金鑰/分配/用量/模型目錄）；儀表板只留總覽；分配/金鑰頁各放一句白話解釋「分配=能用什麼、金鑰=拿來連線的鑰匙」；金鑰卡的「改名」+「編輯 model」合成單一「編輯」；admin Provider 頁英文「Rotate」中文化。**純前端，無 schema 變更。**

## 背景與問題

會員目前只有單一「我的儀表板」頁，一路長捲塞了七種東西：身份、API 端點、安裝 Codex、我的金鑰、用量圖、可自助領取的 model、我的分配、上手引導。**這塊沒被真正設計過**——資訊混在一起，非技術成員難找到「我要做的事」在哪。

會員其實只有三件事：**怎麼連（金鑰）、我能用什麼（分配）、花了多少（用量）**。把這三件事各自給一個明確的地方，儀表板只留「一眼看懂」的總覽，並用一句話講清「分配 vs 金鑰」，就能大幅降低混亂。對應原則 6 可達性（白話 UX、降低進入門檻）與原則 5 集中管理（每件事有單一所在地）。

同時順手收兩個 UI 小問題：金鑰卡的「改名」「編輯 model」是同一把金鑰的編輯、不該拆兩顆按鈕；admin Provider 憑證頁的英文「Rotate」與會員端「重新產生」用詞不一致。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 會員分頁導覽（Priority: P1）🎯 MVP

會員從頂部導覽就能直達「金鑰 / 分配 / 用量」三件事，各自一頁、互不混雜；儀表板只顯示總覽。

**Why this priority**: 把「一頁長捲」拆成清楚的去處，是整個重設計的核心；其餘建立在分頁結構上。

**Independent Test**: 以會員身分登入 → 頂部看到「我的儀表板 / 金鑰 / 分配 / 用量 / 模型目錄」→ 點各分頁載入對應內容、網址可深連結與重新整理。

**Acceptance Scenarios**:

1. **Given** 會員已登入，**When** 檢視頂部導覽，**Then** 出現「我的儀表板 / 金鑰 / 分配 / 用量 / 模型目錄」（admin 額外項不變）。
2. **Given** 任一分頁，**When** 直接以其網址開啟或重新整理，**Then** 正確載入該頁（深連結可用）。
3. **Given** 舊的儀表板深連結（如單筆分配詳情），**When** 開啟，**Then** 仍可達（保留路由或導向新位置）。

---

### User Story 2 - 精簡儀表板（總覽）（Priority: P1）

儀表板只放「一眼看懂」的摘要：本月用量/花費對配額、活躍金鑰數與分配數、一個快速接入（安裝 Codex 一行指令）、待辦提示（沒金鑰→去建、有可領取 model→去領）。

**Why this priority**: 沒有精簡的總覽，分頁就只是把長捲搬家；總覽是會員「我現在狀況如何」的著陸點。

**Independent Test**: 開儀表板 → 只看到摘要區塊（用量摘要 + 金鑰/分配計數 + 快速接入 + 待辦），不再有金鑰卡全文 / 完整圖表 / 分配卡全列。

**Acceptance Scenarios**:

1. **Given** 會員，**When** 開儀表板，**Then** 看到本月用量/花費對配額、活躍金鑰數、活躍分配數、安裝 Codex 一行指令。
2. **Given** 會員還沒有任何金鑰，**When** 開儀表板，**Then** 待辦提示「去建立第一把金鑰」並連到金鑰頁。
3. **Given** 會員有可自助領取的 model，**When** 開儀表板，**Then** 待辦提示「有可領取的 model」並連到分配頁。
4. **Given** 各摘要，**When** 點計數 / 摘要，**Then** 連到對應分頁（金鑰 / 分配 / 用量）。

---

### User Story 3 - 分配 vs 金鑰 一句話講清（Priority: P1）

分配頁與金鑰頁各放一句白話解釋，讓會員秒懂兩者關係。

**Why this priority**: 「分配」「金鑰」兩個詞並存，沒解釋就是混亂源；一句話成本極低、效益高。

**Independent Test**: 開分配頁與金鑰頁 → 各看到「分配＝你能用哪些模型；金鑰＝拿來連線的鑰匙」這句（或等義）。

**Acceptance Scenarios**:

1. **Given** 分配頁，**When** 檢視頁首，**Then** 看到「分配＝你能用哪些模型；金鑰＝拿來連線的鑰匙」說明。
2. **Given** 金鑰頁，**When** 檢視頁首，**Then** 看到同一句（或對稱）說明。

---

### User Story 4 - 金鑰卡「編輯」合一 + Rotate 用詞統一（Priority: P2）

金鑰卡把「改名」與「編輯 model」合成單一「編輯」對話框（名稱 + model 多選一起存）；admin Provider 憑證頁的英文「Rotate」改成與會員一致的中文。

**Why this priority**: 兩個 UI 一致性收尾；非阻斷，但留著會持續造成小困惑。

**Independent Test**: 金鑰卡只剩單一「編輯」（點開可同時改名與改 model）；admin Provider 頁無英文「Rotate」字樣。

**Acceptance Scenarios**:

1. **Given** 金鑰卡某把，**When** 點「編輯」，**Then** 同一對話框可改名稱與可用 model，一起儲存；外面不再有獨立「改名」「編輯 model」兩顆。
2. **Given** 金鑰卡，**When** 檢視動作，**Then** 「重新產生」「撤回」仍為獨立、不被收進編輯框。
3. **Given** admin Provider 憑證頁，**When** 檢視動作，**Then** 原英文「Rotate」已中文化（與會員「重新產生」一致）；功能不變。

---

### Edge Cases

- **空狀態**：沒有金鑰 / 沒有分配 / 沒有用量 → 各分頁與儀表板待辦顯示明確空狀態與下一步，不顯示誤導數字。
- **深連結 / 書籤**：既有 `/dashboard/...` 連結要可達（保留或 redirect）。
- **手機（360px）**：頂部分頁列要沿用既有手機收合 / 橫向捲動，不溢出。
- **admin 也是會員**：admin 仍看得到會員分頁，admin 專屬項另外呈現、不衝突。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 會員頂部導覽 MUST 提供 `我的儀表板 / 金鑰 / 分配 / 用量 / 模型目錄`；各為可深連結、可重新整理的獨立位置。
- **FR-002**: **儀表板（總覽）** MUST 只顯示摘要：本月用量/花費對配額、活躍金鑰數、活躍分配數、一個快速接入（安裝 Codex 一行指令）、待辦提示；MUST **不**在儀表板顯示金鑰卡全文 / 完整用量圖表 / 分配卡全列。
- **FR-003**: **金鑰頁** MUST 含應用金鑰管理（建立/編輯/重新產生/撤回）、API 端點、如何呼叫、安裝 Codex。
- **FR-004**: **分配頁** MUST 含分配卡（model + 配額/用量）、可自助領取、進單筆分配詳情。
- **FR-005**: **用量頁** MUST 含用量摘要 + 圖表（趨勢 + 各 model + 熱度圖）。
- **FR-006**: 分配頁與金鑰頁 MUST 各顯示一句白話解釋「分配＝能用什麼；金鑰＝拿來連線的鑰匙」。
- **FR-007**: 儀表板待辦提示 MUST 正確反映狀態（無金鑰→提示建立並連金鑰頁；有可領取 model→提示領取並連分配頁）。
- **FR-008**: 金鑰卡 MUST 把「改名」與「編輯 model」合成**單一「編輯」**（同對話框改名稱 + 可用 model）；「重新產生」「撤回」維持獨立。
- **FR-009**: admin Provider 憑證頁的「Rotate」 MUST 中文化（與會員「重新產生」一致）；功能不變。
- **FR-010**: 既有後端 / 用量計算 / proxy / 既有會員深連結 MUST 零回歸（本功能純前端、無 schema、無新端點）。

### Key Entities *(include if feature involves data)*

- 無新增資料實體。沿用既有「分配（allocation）」「應用金鑰（credential）」「用量（call records）」的呈現；本階段只重排版位與用語、不改語意。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 會員頂部導覽有 5 個去處（儀表板/金鑰/分配/用量/模型目錄）；每個分頁可直接以網址開啟並正確載入。
- **SC-002**: 儀表板只顯示總覽（摘要 + 計數 + 快速接入 + 待辦）；不再混入金鑰卡全文 / 完整圖表 / 分配卡全列。
- **SC-003**: 分配頁與金鑰頁各有「分配 vs 金鑰」一句解釋。
- **SC-004**: 金鑰卡的「改名 + 編輯 model」已合成單一「編輯」；admin Provider 頁無英文「Rotate」。
- **SC-005**: 既有會員深連結（含單筆分配詳情）100% 仍可達；既有後端 / 用量 / proxy 零回歸。
- **SC-006**: 桌機 + 360px 手機皆不溢出（沿用階段 16 RWD）。

## Assumptions

- **導覽用頂部分頁（獨立路由頁）**，非 dashboard 內 tabs（會員已選定）。各分頁是獨立 URL，便於深連結與書籤。
- **「用量」獨立成一頁**（總覽只放小摘要，完整圖表在用量頁）。
- **保留「分配」「金鑰」兩詞**（不改用「我可用的模型」），以一句解釋補白話。
- **admin 專屬項**（既有 admin 導覽）不變；admin 仍可看會員分頁。
- **純前端**：把既有元件（`app-credentials-card`、分配卡、`UsageSummary`/圖表、`CodexInstallCard`、`AllocationKeysReadonly` 等）搬進各自路由；新增總覽元件；無新端點、無 schema、無 migration。
- **深連結相容**：既有 `/dashboard/...` 路由保留或 redirect。
- **平台 / 前端 stack** 沿用既有（React/Vite + shadcn/ui；既有頂部 + 手機收合導覽）。
