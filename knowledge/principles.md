# 原則

## 根公理

**分享就是資源的分配。**

在組織內部脈絡下，AI API 的存取權不是抽象的「許可」，而是具體的「資源」
——包含模型存取、用量配額、速率上限、成本歸屬。分享一個 API 就是把資源
的一部分分配給某個團隊或個人；分配是有對象、有額度、可調整、可收回的，
不是一次性的移交。

> 工程通則（TDD、契約優先、可觀測性、YAGNI 等）見 `.specify/memory/constitution.md`。
> 本檔僅記錄「AI API 分享與權限管理」的領域原則。

## 衍生原則

### 1. 憑證隔離 (Credential Isolation)

- **推導自**：根公理（每筆分配都是獨立的領取，不能共用單據）
- **意義**：被分配者永遠拿不到底層 AI 供應商的 API key；系統為每一筆分配
  發行獨立、可識別、可撤回的憑證。
- **如何應用**：
  - 所有對外發出的 token 必須綁定到一筆分配記錄
  - 底層供應商 key 僅存在於服務內部，絕不出現在 API 回應、日誌、錯誤訊息中
  - 撤銷單一憑證不影響其他憑證

### 2. 可追蹤性 (Traceability)

- **推導自**：根公理（分配本質要求帳目清楚）
- **意義**：每一筆分配必須綁定到可識別的資源（模型／端點／專案），且每一次
  API 呼叫都可追溯到「分配 → 被分配者 → 團隊」。資源端與使用端兩面都可追蹤，
  才構成完整審計能力。
- **如何應用**：
  - 分配記錄必有資源綁定欄位；不允許「萬用分配」存在
  - 呼叫日誌必帶分配 ID 與身份標記
  - 不存在匿名共用憑證；每次呼叫都歸屬到具體分配對象
  - 範圍／配額／過期為可選的加強控制，非強制欄位

### 3. 即時撤回 (Instant Revocation)

- **推導自**：根公理（分配可以收回；資源不是送出去就拿不回來的）
- **意義**：擁有者撤回分配後，後續任何呼叫必須立刻被拒絕，不依賴 token 自然
  過期。
- **如何應用**：
  - 每次呼叫驗證分配的「當前狀態」，而不只是 token 簽章
  - 撤回操作生效時間需定義 SLO（例：N 秒內全節點可見）
  - 撤回必留稽核紀錄（誰、何時、為何撤回）

### 4. 轉分配需顯式允許 (No Implicit Re-allocation)

- **推導自**：根公理（被分配到資源的人，預設沒有再分配的權力）
- **意義**：被分配者預設**不能**再把資源分配給第三方；如需轉分配，必須由
  擁有者明確開啟此能力。
- **如何應用**（當轉分配功能實作時）：
  - 分配記錄上有 `can_redelegate` 旗標，預設為 `false`
  - 即使開啟轉分配，二次分配的範圍不得超過原分配；且擁有者仍可見、可撤回
    整條分配鏈
