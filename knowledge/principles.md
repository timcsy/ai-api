# 原則

## 根公理

**分享就是資源的分配。**

在組織內部脈絡下，AI API 的存取權不是抽象的「許可」，而是具體的「資源」
——包含模型存取、用量配額、速率上限、成本歸屬。分享一個 API 就是把資源
的一部分分配給某個團隊或個人；分配是有對象、有額度、可調整、可收回的，
不是一次性的移交。

> 工程通則（TDD、契約優先、可觀測性、YAGNI 等）見 `.specify/memory/constitution.md`。
> 本檔僅記錄「AI API 分享與權限管理」的領域原則。

## 衍生原則

### 1. 憑證隔離 (Credential Isolation)

- **推導自**：根公理（每筆分配都是獨立的領取，不能共用單據）
- **意義**：被分配者永遠拿不到底層 AI 供應商的 API key；系統發行獨立、可識別、可撤回的憑證。
  **唯一性與額度的單位是「分配」而非「token」**——憑證是一把**可命名的應用 key（scoped application
  credential）**，其 scope 為**一組分配**（≥1）；每次呼叫**依 request 的 model 歸戶到對應分配**。額度、
  歸戶、可追蹤性皆綁在**分配**層，憑證只是「能用哪些分配」的存取方式。〔業界對照：scoped API key /
  service account / OAuth scopes；per-call metered to the matching allocation。M:N 實作見願景階段 20，
  階段 18 的「一憑證綁一分配」是其 1:N 特例。〕
- **如何應用**：
  - 所有對外發出的 token 必須綁定到**一組分配**（≥1 筆分配記錄）；同一把 key 內各分配的 model **不得重複**
    （否則歸戶有歧義）
  - 底層供應商 key 僅存在於服務內部，絕不出現在 API 回應、日誌、錯誤訊息中
  - **一個分配可同時被多把憑證授權，一把憑證也可授權多筆分配（per-device / per-application）**；
    新增/撤銷單把不影響其他（「撤銷單一憑證不影響其他憑證」名副其實）
  - **額度與用量歸戶在分配層**；token 數量不繞過分配的配額與異常偵測（多把 token 共用各自分配的額度）
    → 故無需對 token 數量設限
  - **attenuation（不提權）**：一把 key 的 scope 只能含**擁有者自己已被授予的分配**；打包既有授權、不創造新權限
  - 同一 model 可有多個分配（不同用途／方式），各自獨立額度與身分

### 2. 可追蹤性 (Traceability)

- **推導自**：根公理（分配本質要求帳目清楚）
- **意義**：每一筆分配必須綁定到可識別的資源（模型／端點／專案），且每一次
  API 呼叫都可追溯到「分配 → 被分配者 → 團隊」。資源端與使用端兩面都可追蹤，
  才構成完整審計能力。
- **如何應用**：
  - 分配記錄必有資源綁定欄位；不允許「萬用分配」存在
  - 呼叫日誌必帶分配 ID 與身份標記
  - 不存在匿名共用憑證；每次呼叫都歸屬到具體分配對象
  - 範圍／配額／過期為可選的加強控制，非強制欄位

### 3. 即時撤回 (Instant Revocation)

- **推導自**：根公理（分配可以收回；資源不是送出去就拿不回來的）
- **意義**：擁有者撤回分配後，後續任何呼叫必須立刻被拒絕，不依賴 token 自然
  過期。
- **如何應用**：
  - 每次呼叫驗證分配的「當前狀態」，而不只是 token 簽章
  - 撤回操作生效時間需定義 SLO（例：N 秒內全節點可見）
  - 撤回必留稽核紀錄（誰、何時、為何撤回）

### 4. 轉分配需顯式允許 (No Implicit Re-allocation)

- **推導自**：根公理（被分配到資源的人，預設沒有再分配的權力）
- **意義**：被分配者預設**不能**再把資源分配給第三方；如需轉分配，必須由
  擁有者明確開啟此能力。
- **如何應用**（當轉分配功能實作時）：
  - 分配記錄上有 `can_redelegate` 旗標，預設為 `false`
  - 即使開啟轉分配，二次分配的範圍不得超過原分配；且擁有者仍可見、可撤回
    整條分配鏈

### 5. 集中管理勝於並行管理 (Single Authority for Access)

- **推導自**：根公理 + 可追蹤性（多條路徑同時管同一件事 → 必然 drift →
  追蹤斷裂 → 違反可追蹤性）
- **意義**：對「誰能存取什麼」這類治理決策，系統內**只有一條唯一路徑**可改寫；
  其他機制要嘛 derive 自它、要嘛只在它不存在時生效（bootstrap fallback）。
  並行管理路徑會逼著管理員兩邊同步、又遲早不同步，使用者被無聲擋下。
- **如何應用**：
  - 成員清單是 admin mode 下 access 的單一真理；email 白名單退為
    bootstrap-only（DB 無 admin 才查）
  - 一個語意旗標支援多個治理用途（例：分配上的 service flag 同時豁免
    rebalance 與 anomaly），不為每個治理機制各加專用 exempt 旗標
  - infra 類設定（body size 等）：唯一真理在 Helm value，UI 可顯示、不可編輯

### 6. 可達性 (Accessibility) — 被分配者必須能實際取用

- **推導自**：根公理（分配的目的是讓對方**真的用到**資源；若被分配者因技術門檻、
  缺對應 UI、或裝置限制而無法實際取用，這筆分配只是名義上的——不算完成的分配）
- **意義**：資源不只是「發出去」，還要確保被分配的對象（含不會寫程式的成員、用手機的人）
  能在**不依賴工程師**的情況下實際使用與自助管理。可達性是分配是否真正成立的一部分，
  不是事後 polish。
- **如何應用**：
  - 面向非技術使用者用白話 UX；admin 與成員的每項能力都要有對應 UI——
    「後端有 API 但只能靠工程師 / SQL 才能操作」視為**功能未完成**，非「待 polish」
  - 自助操作不需另一個 actor 協助（成員自助領憑證 / 看用量總覽、admin 自助設定通知）
  - 跨裝置可用（桌機完整、手機堪用且順）；以「使用情境目錄」降低不熟 LLM API 者的進入門檻

### 7. 演進性 (Evolvability) — 核心穩定、邊緣快變，用適配層隔開

- **推導自**：根公理（分配的價值在「**誰能用什麼、可計量、可撤回、可追蹤**」——這層與 AI 能力的
  快速更迭無關。AI 變的是邊緣，不是這個價值核心；故核心必須與快變的 AI 邊緣**隔離**，否則邊緣的
  churn 會把核心一起拖垮、專案隨 AI 變遷而腐化。）
- **意義**：AI 變的是**邊緣**（有哪些模型、哪家 API 的形狀、哪個能力旗標、哪個客戶端工具），
  不是**核心**（分配 / 憑證 / 計費 / 存取）。長期可維護＝讓快變鎖在邊緣、慢變留在核心，中間以
  薄適配層隔開；新能力以「**資料 + 實測**」承載，而非烤進程式。
- **如何應用**：
  - **適配層**：快速演進的外部依賴（litellm 等）只透過**單一 adapter** 對話
    （`proxy/upstream.py`、`services/litellm_registry.py`）——版本變動只改一處，爆炸半徑鎖在邊緣。
  - **資料勝於程式**：易變的東西（模型 / 價格 / 能力旗標）做成**可同步、可編輯的資料**，不 hardcode；
    「新模型上市」＝加資料 / 重新同步，不是改碼重部署。
  - **實測勝於臆測**：能力不確定或會變的，做成「**可測試 + admin 可覆寫**」（responses 雙來源、依種類測模型），
    別把「模型 X 支援 Y」烤進靜態能力表——硬編會過時，「打一次就知道」永遠成立。
  - **註冊表**：同類但會增加的東西（應用、provider、使用情境）用**註冊表**，讓「新增」＝加一筆資料而非改架構。
  - **守住軸的正交**：新功能先問「**這是哪條軸**」（①模型原生 API 型態 ②模型能力 ③gateway 端點可用性
    ④客戶端工具…），**別 overload 既有欄位**（把 responses 從 mode 推導出來的 latent bug 即代價）。
  - **用迴圈安全吸收變化**：變化先過原則（不服務原則就先不做）→ build-vs-adopt 在 specify 之前定 →
    採用前先驗證能力邊界（`inspect.signature` / 印一次回傳值）→ TDD + CI 綠 + 部署煙霧 → 分鐘級回滾；
    每次踩坑蒸餾進 `experience.md`（讓專案越維護越聰明，但要定期蒸餾、防知識腐化）。
